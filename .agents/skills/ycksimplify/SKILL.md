# 简化：代码审查与清理

审查所有已更改的文件，检查代码复用性、质量和效率。修复发现的任何问题。

## 第一阶段：识别更改内容

运行 `git diff`（如果存在暂存更改，则运行 `git diff HEAD`）来查看具体改动。如果没有 git 更改，则审查用户提到的、或你在本次对话中较早编辑过的最近修改的文件。

## 第二阶段：并行启动三个审查 Agent

使用 ${AGENT_TOOL_NAME} 工具，在单条消息中同时启动全部三个 Agent。给每个 Agent 传递完整的 diff，以便它们拥有完整上下文。

### Agent 1：代码复用审查

针对每一处改动：

1. **搜索现有的工具函数和辅助函数**，看看能否替换新写的代码。在整个代码库中查找相似模式——常见位置包括工具目录、共享模块，以及与更改文件相邻的文件。
2. **标记任何与现有功能重复的新函数**。建议直接使用已有的函数代替。
3. **标记任何可以直接调用现有工具的内联逻辑**——手写的字符串处理、手动路径处理、自定义环境检查、临时类型守卫等都是常见候选对象。

### Agent 2：代码质量审查

针对同一批改动，检查以下“hacky”模式：

1. **冗余状态**：与现有状态重复的状态、可直接派生的缓存值、原本可以直接调用的观察者/副作用
2. **参数膨胀**：给函数新增参数，而不是对现有参数进行泛化或重构
3. **轻微变动的复制粘贴**：几乎相同的代码块，应通过共享抽象进行统一
4. **泄漏的抽象**：暴露了本应封装的内部细节，或破坏了现有的抽象边界
5. **字符串类型滥用**：使用原始字符串，而代码库中已有常量、枚举（字符串联合类型）或品牌类型
6. **不必要的 JSX 嵌套**：没有实际布局价值的包装 Box/元素——检查内部组件的 props（如 flexShrink、alignItems 等）是否已能满足需求
7. **嵌套条件**：三层以上的三元表达式链（a ? x : b ? y : ...）、嵌套 if/else 或嵌套 switch——应使用提前返回、守卫子句、查找表或 if/else-if 级联来扁平化
8. **不必要的注释**：解释代码“做什么”的注释（好的命名已经足够）、叙述改动内容、引用任务或调用者的注释——全部删除；仅保留非显而易见的“为什么”（隐藏约束、微妙不变量、临时方案）

### Agent 3：效率审查

针对同一批改动，检查效率问题：

1. **不必要的工作**：冗余计算、重复的文件读取、重复的网络/API 调用、N+1 查询模式
2. **错过的并发**：相互独立的操作被串行执行，而本可以并行运行
3. **热路径膨胀**：在启动阶段或每个请求/每帧渲染的热路径上新增了阻塞性工作
4. **反复的无用更新**：在轮询循环、定时器或事件处理器中无条件进行的状态/Store 更新——应添加变更检测守卫，避免下游消费者收到无意义通知。同时：如果包装函数接受 updater/reducer 回调，请确认它会正确处理“相同引用返回”（或对应的“无变更”信号）——否则调用者的提前返回无用逻辑会被默默失效
5. **不必要的存在性检查**：在操作前先检查文件/资源是否存在（TOCTOU 反模式）——应直接操作并处理错误
6. **内存问题**：无界数据结构、缺少清理、事件监听器泄漏
7. **过于宽泛的操作**：读取整个文件而只需其中一部分、加载全部条目而只需过滤一个

## 第三阶段：修复问题

等待三个 Agent 全部完成。汇总它们的发现，并直接修复每个问题。如果某个发现是误报或不值得处理，请注明并跳过——不要与发现争辩，直接忽略即可。

完成后，简要总结修复了什么（或确认代码原本就很干净）。
