# AI 产品经理求职路线图

## 定位

Reo 的求职主线是 AI 产品经理，增长产品经理作为副线证明，FDE 式用户现场与部署理解作为新的加分能力层，前端部署和工程协作能力作为支撑项。

Reo 不是为了面试而做完整商业化 SaaS。Reo 要证明候选人能把 AI 能力变成可用、可评估、可增长、可商业化的产品。

Reo 的产品差异化是把记录变成 agent 可使用的创作上下文。记忆空间不是静态资料夹，而是可以被 Codex-style agent 使用的 workspace；Reo 通过初始化 `AGENTS.md`、skills、plugins 和任务模板，让普通用户也能用自己的记录生成可追溯的 PRD、增长实验、报告和设计系统约束 HTML artifact 文件；隔离预览能力完成前，Reo trusted renderer 不执行、不注入、不渲染用户 HTML。

AI 产品经理的证明方式不再只是写 PRD。Reo 要展示 AI-native PM 工作方式：短周期 demo、eval-first、内测反馈、模型能力检查、简单实现和快速 shipping。

第一阶段只服务一个主 ICP：需要整理用户访谈、会议复盘或研究材料的早期产品经理。创业者、研究型 PM、独立产品人和转行 AI PM 可以作为扩展画像，不进入第一版主叙事。

## 面试证明目标

Reo 必须能回答这些面试问题：

- 目标用户是谁，为什么这个问题值得做。
- AI 能力如何进入真实工作流，而不是停留在聊天框。
- AI 输出质量如何评估、迭代和降风险。
- 本地优先、隐私、成本、延迟和商业化之间如何取舍。
- 增长漏斗在哪里，如何验证获客、激活和留存。
- 如何进入用户真实工作流，把现场反馈转成可复用产品模式。
- 如何设计人机协作边界、AI 自主度、用户确认点和失败恢复。
- 如何把长期记录变成 agent 可使用的创作上下文。
- 如何让普通用户在设计系统和 workspace 规则约束下使用 agent 创作。
- 哪些能力必须先做，哪些商业化基础设施可以后置。

## 最高性价比交付物

| 优先级 | 交付物                        | 面试证明                                                                 |
| ------ | ----------------------------- | ------------------------------------------------------------------------ |
| P0     | 可演示 AI 工作流              | AI 产品定义、场景理解、交互设计、工程协作                                |
| P0     | AI eval 与成本评估            | 模型质量意识、指标体系、上线风险控制                                     |
| P0     | 产品 case study               | 用户洞察、取舍、路线图、跨职能表达                                       |
| P1     | 内测群与反馈闭环              | 用户运营、用户研究、需求优先级、留存验证                                 |
| P1     | 增长漏斗与指标树              | Growth PM 副线、数据意识、实验设计                                       |
| P1     | FDE 式 field loop             | 用户现场理解、快速原型、部署边界、pattern codification，合并进内测反馈包 |
| P1     | 5 分钟演示视频                | 表达能力、产品叙事、面试效率                                             |
| P2     | 公开 landing page 或 waitlist | 获客验证、SEO 起点、商业化意识                                           |
| P2     | 商业化基础设施蓝图            | 会员、计费、云资源和合规认知                                             |

只要某个交付物不能提升 AI 产品经理面试通过率，就写进 case study 或 public release gate，不进入第一批开发。

主交付物最多 5 个。AI-native PM operating model、human-AI collaboration map、FDE-shaped field loop、model release review 只能作为 demo、eval、内测包或 case study 的组成部分，不单独膨胀成主交付物。

## 核心产品切片

第一版最适合面试的切片是：

```text
导入或录制用户访谈/会议材料
  -> 转写或整理为片段
  -> AI 提取需求、痛点、机会点和证据引用
  -> 生成 PRD 草案、用户故事或增长实验假设
  -> 用户回到本地记忆空间继续编辑和沉淀
```

这个切片同时展示 AI PM 和 Growth PM 能力：

- AI PM：定义 AI 工作流、输出结构、证据引用、失败恢复、质量评估。
- Growth PM：从用户访谈中提取增长假设、激活阻力、转化机会和实验优先级。

## 不优先做的内容

这些能力暂时只保留规划，不进入求职第一批开发：

- 真实支付、退款、发票和完整会员系统。
- 手机号验证码、微信登录和账号合并。
- 阿里云生产部署、备案、CDN、OSS、SLS 和完整监控。
- 完整运营后台、CMS 和客服工具。
- 完整 AI Gateway、多模型路由和 OpenMeter/Lago 生产集成。

这些内容能证明商业化认知，但第一阶段不会比一个强 AI 产品切片更能提升面试成功率。

## 阶段路线

### Career Phase 0：作品集定位

输出：

- 一句话定位：Reo 是本地优先的 AI 产品工作台，用于从非结构化材料中提取可追溯产品洞察。
- 目标用户：需要整理用户访谈、会议复盘或研究材料的早期产品经理。
- 使用场景：用户访谈、会议复盘、竞品研究、需求整理、增长假设生成。
- 不做清单：完整 SaaS、企业协作、真实支付、完整后台。

完成标准：

- 面试时能在 60 秒内讲清 Reo 的用户、问题、方案和取舍。
- 能解释为什么没有先做完整商业化基建。

### Career Phase 1：可演示 AI 工作流

输出：

- 一个稳定可跑的核心流程。
- 输出结构包含需求、痛点、证据引用、风险和下一步建议。
- 对失败、空输入、低质量转写和敏感内容有可见恢复路径。
- Demo 样本包含稳定成功样本、真实噪音样本和失败样本。

完成标准：

- 5 分钟 demo 能从输入材料走到结构化产品洞察。

### Career Phase 2：AI 质量评估

输出：

- 20 到 30 条样本材料。
- 质量维度：准确性、完整性、证据引用、幻觉率、可操作性、延迟、成本。
- 对比表：人工整理、普通聊天式 AI、Reo 工作流。
- 成本表包含输入规模、调用次数、单次成本、平均延迟、失败率和人工修正时间。

完成标准：

- 能解释 Reo 如何判断 AI 输出是否可用，而不是只展示一次成功 demo。

### Career Phase 3：产品 case study

输出：

- 背景、用户、问题、竞品、取舍、核心切片、指标、风险、路线图。
- 两张关键图：用户旅程图、指标树。
- 一页商业化延展：会员、成本治理、登录、支付、SEO 和遥测为什么后置。
- 一页工程协作延展：本地优先、服务端密钥边界、AI provider 调用、发布风险和验收标准。

完成标准：

- 面试官不看代码也能判断候选人具备产品思维。

### Career Phase 4：增长副线

输出：

- 内测群运营方案：招募画像、入群问卷、任务脚本、反馈模板、周节奏、退出标准。
- Landing page 或 waitlist 草案。
- 激活漏斗：访问、下载、创建工作区、完成首个 AI 洞察、7 日内回访。
- 增长实验 backlog：定位文案、样例模板、首次任务引导、价格页假门。
- 用户反馈闭环：反馈收集、问题归类、证据链接、优先级决策、版本回访。

完成标准：

- 能把 Reo 讲成有真实用户验证、增长假设和反馈闭环的产品，而不是个人工具。

### Career Phase 5：面试包

输出：

- 5 分钟演示视频。
- 8 到 12 页 case study。
- 简历项目 bullet。
- 面试讲稿：AI PM 主线、Growth PM 副线、商业化基础设施后续计划。
- Appendix：AI-native PM operating model、FDE-shaped field loop、model release review 和商业化 release gate。

完成标准：

- 5 个主交付物都能映射到岗位能力：用户洞察、AI 产品设计、指标、商业化、跨职能协作。

## 商业化与求职的关系

商业化基础设施仍然保留为长期 initiative，但求职阶段只实现能增强作品集可信度的部分。

| 商业化能力                  | 求职阶段处理                                         |
| --------------------------- | ---------------------------------------------------- |
| 会员、支付、登录            | 写入 public release gate，不进入第一批开发           |
| AI Gateway、火山消费治理    | 先做成本模型和服务端边界设计，后续按 spec 实现       |
| SEO、landing page、waitlist | 可以作为增长副线第一批展示                           |
| 遥测                        | 先定义事件字典和指标树，真实采集后置                 |
| 后台                        | 用 case study 截图或低保真图表达，不做完整后台       |
| 内测群运营                  | 求职阶段优先做，用于验证问题、收集反馈和沉淀 PM 证据 |

## 面试叙事

推荐叙事顺序：

1. 我没有把 Reo 做成大而全 SaaS，因为转行阶段最重要的是证明 AI 产品判断。
2. 我选择一个高频 PM 场景：从访谈、会议和资料中提取可追溯产品洞察。
3. 我把 Reo 定位成记录到创作的 AI workflow：记忆空间沉淀上下文，Codex-style agent 在 `AGENTS.md`、skills、plugins 和设计系统约束下生成新 artifact。
4. 我设计了本地优先架构，降低隐私风险，并为未来商业化保留服务端边界。
5. 我用 eval 样本评估 AI 输出质量，而不是只依赖主观 demo。
6. 我用内测群做用户招募、任务验证和反馈闭环，把真实反馈转成路线图决策。
7. 我用 FDE 式 field loop 理解用户现场，把一次性反馈沉淀成可复用模板、eval set 和 playbook。
8. 我设计了增长漏斗和 waitlist 验证路径，证明这个产品可以进入市场。
9. 我把会员、支付、登录、云资源和成本治理放入商业化 initiative，等需求和分发信号成立后再做。

## 成功标准

求职阶段完成时，Reo 应该具备：

- 一个稳定的 AI PM demo。
- 一套 AI 输出质量和成本评估材料。
- 一份可公开展示的产品 case study。
- 一个增长漏斗和实验 backlog。
- 一个内测群运营与反馈闭环记录。
- 一个 FDE 式用户现场与部署边界说明。
- 一份商业化基础设施路线图。
- 一套能直接用于简历、作品集和面试的叙事材料。

主叙事只讲：用户真实工作流、快速 demo、eval-first、human-AI collaboration、内测 field signal、路线图取舍。

产品差异化叙事只讲：记录、agent-ready memory space、Codex-style 创作、设计系统约束 artifact、普通用户可用。

## 范围控制

每个候选任务都先回答：

```text
它是否直接提升 AI 产品经理面试通过率？
```

如果答案是否定，处理方式只有两种：

- 写入 case study 的后续计划或 public release gate。
- 删除。
