# 数据

本文档是数据结构与状态归属的当前真源。

## 当前事实

- 当前没有 database schema。
- 当前 durable data contract 是记忆空间文件夹文件和 main-owned memory space registry。
- 当前没有 Drizzle config。
- 当前没有 Better Auth tables。
- 当前没有 auth session persistence owner。
- 当前 TanStack Query key 覆盖记忆空间 snapshot：`['workspace', 'snapshot', workspaceId]`。
- 当前 TanStack Query key 覆盖 Memory detail：`['workspace', 'memory-detail', workspaceId, memoryId]`。
- 当前 TanStack Query key 覆盖 selected Segment content：`['workspace', 'segment-content', workspaceId, memoryId, segmentId]`。
- 当前 TanStack Query key 覆盖 selected SegmentSupplement content：`['workspace', 'segment-supplement-content', workspaceId, memoryId, segmentId, supplementId]`。
- 当前 TanStack Query key 覆盖 memory space list：`['workspace', 'memory-spaces']`。
- 当前 TanStack Query key 覆盖 voice transcription settings：`['settings', 'voice']`。
- 当前没有 Zustand stores。
- 当前补转录不新增 Query key、不新增 durable schema、不新增 Zustand store。自动补转录状态只存在于 main process queue；手动补转录的 running 状态只存在于 App feature-local component state。
- 当前 React Hook Form form owner 覆盖 create memory space submit 前的 title、description、selection token、displayPath draft、Memory create title draft、Memory rename title draft、Segment rename title draft、SegmentSupplement rename title draft 和 memory space rename title draft。
- 当前 Zod runtime schema owner 是 workspace IPC contract 和错误信封。

## 当前实体与字段归属

- 记忆空间是用户选择的本地文件夹；用户内容真源是该文件夹内的普通文件。
- `.reo/workspace.json` 保存 `schemaVersion`、`workspaceId`、title、description 和 createdAt；它定义记忆空间身份，不保存 UI 状态、root path、handle 或查询缓存。
- Workspace snapshot 是 IPC response 和 TanStack Query cache 投影，只包含 `workspaceId`、title、description 和 `memories[]`；不包含顶层 `segments[]`、root path、selection token 或 `workspaceHandle`。
- 记忆空间 root 是 Workspace 文件空间根；`.reo/workspace.json` 保存 Workspace stable id 和 title。Memory、Segment 和 SegmentSupplement 是用户内容文件空间节点：用户可编辑语义写在 Markdown/frontmatter，稳定 id、归属、时长、音频字节数、正文 byte 长度和事务字段写在 `.reo/objects/<kind>/<id>.json` manifest 中；目录 basename 是用户可见名称真源；Markdown frontmatter title 是 Reo 持锁写入的语义镜像。目录 basename 等于稳定 id 时使用 Markdown title 投影；目录 basename 为 `<id>--<title>` 或用户手动改名后的安全名称时使用 basename title 投影。
- Memory summary 由 `.reo/index.json` 的 `memories[]` 提供，字段包含 `memoryId`、title、createdAt、updatedAt、segmentCount、audioSegmentCount、noteSegmentCount、audioDurationMs、audioByteLength、hasAudioTranscript、hasAnyNote 和 supplementCount；`hasAnyNote` 表示存在 note Segment 或 note SegmentSupplement。
- Memory detail 由 matching Memory 目录内的 `memory.md`、`.reo/objects/memories/<memoryId>.json` 和该 Memory 内 finalized Segment 文件空间节点派生，字段包含 `workspaceId`、Memory summary 字段和 `segments[]`。`segments[]` 投影 finalized `audio` 和 `note` Segment；audio Segment 字段包含 `workspaceId`、`memoryId`、`segmentId`、`type: 'audio'`、title、可选 `contentTitle`、createdAt、updatedAt、durationMs、audioByteLength、`transcript.exists`、`lastTranscriptionAttempt`、supplementCount、`supplements[]` 和 `contentTabOrder`；note Segment 字段包含 `workspaceId`、`memoryId`、`segmentId`、`type: 'note'`、title、可选 `contentTitle`、createdAt、updatedAt、bodyByteLength、supplementCount、`supplements[]` 和 `contentTabOrder`。Segment primary content tab 的用户可见名称来自 `segment.md` frontmatter `content_title`，缺失时由 renderer 按 Segment type 显示 `转录` 或 `正文`。
- 每个 Segment projection 内的 `supplements[]` 投影 finalized `audio` 和 `note` SegmentSupplement；audio SegmentSupplement 字段包含 `workspaceId`、`memoryId`、`segmentId`、`supplementId`、`type: 'audio'`、title、createdAt、updatedAt、durationMs、audioByteLength、`transcript.exists` 和 `lastTranscriptionAttempt`；note SegmentSupplement 字段包含 `workspaceId`、`memoryId`、`segmentId`、`supplementId`、`type: 'note'`、title、createdAt、updatedAt 和 bodyByteLength。

- Memory 的 Segment 关系由该 Memory 目录下合法 finalized Segment 文件空间节点表达：`segments/*/segment.md` 必须有合法 frontmatter，`.reo/objects/segments/<segmentId>.json` 的 `memoryId` 必须匹配父 Memory manifest，`segmentId` 必须匹配 Segment directory id；audio Segment 必须满足 audio 文件与 manifest 投影一致，note Segment 必须满足 Markdown 正文 byte 长度与 manifest `bodyByteLength` 一致。当前不保存显式 Segment id 列表；关系来自合法文件空间节点和 `.reo/objects/*` manifest。当前 finalized read model 只接受 `audio` 和 `note`；photo、video、imported_file 和其它 kind 不进入投影。
- Audio segment draft 属于 `.reo/drafts/segments/<segmentId>/`，并继续使用 draft-local `segment.json` 与 `audio.webm` 表达未完成事务；finalized audio segment 属于 matching Memory 的 `segments/<segmentDirectory>/`，并通过 `segment.md`、`.reo/objects/segments/<segmentId>.json` 和 `audio.webm` 表达该段主体记录。Note Segment draft 同样属于 `.reo/drafts/segments/<segmentId>/`，并使用 draft-local `segment.json` 与 `segment.md` body 表达未完成事务。Audio SegmentSupplement draft 属于 `.reo/drafts/supplements/<supplementId>/`，并通过 draft-local `supplement.json` 与 `audio.webm` 表达 selected Segment 的补充录音草稿；Note SegmentSupplement draft 同样属于 `.reo/drafts/supplements/<supplementId>/`，并通过 draft-local `supplement.json` 与 `supplement.md` body 表达 selected Segment 的补充笔记草稿。Finalized audio SegmentSupplement 属于 matching Segment 的 `supplements/<supplementDirectory>/`，并通过 `supplement.md`、`.reo/objects/supplements/<supplementId>.json` 和 `audio.webm` 表达该段主体记录的补充内容。
- Finalized audio Segment 的 `.reo/objects/segments/<segmentId>.json` 必须与所在 memory directory 和 segment directory 的 object id 一致；`memoryId` 与 `segmentId` 不由目录 basename 直接推断。Finalized Segment manifest 的 `contentTabOrder` 是 Reo 管理的可选 durable presentation 字段，元素为 `segment` 或 `supplement:<supplementId>`；读取投影时按当前合法内容 tab 归一化，忽略 stale id，并把缺失的 live tab 追加到末尾。Segment primary content title 不写入 manifest；它属于 `segment.md` frontmatter `content_title`。Finalized audio Segment 与 SegmentSupplement manifest 的 `lastTranscriptionAttempt` 是 Reo 管理的可选技术状态字段，取值为 `'success'`、`'failed'` 或 `'never'`；读取投影时字段缺失按 `'never'` 投影。手动补转录 mode 不写入 manifest，不扩展该字段取值。
- Finalized note Segment 的 `.reo/objects/segments/<segmentId>.json` 必须与所在 memory directory 和 segment directory 的 object id 一致，manifest `kind` 必须为 `"note"`，并写入 `bodyByteLength`；Finalized note SegmentSupplement 的 `.reo/objects/supplements/<supplementId>.json` 必须与所在 SegmentSupplement directory 的 object id 一致，manifest `kind` 必须为 `"note"`，并写入 `bodyByteLength`。Note projection 不暴露 duration、audio byte、transcript 或 lastTranscriptionAttempt 字段。Note Segment 和 note SegmentSupplement 可以包含 `attachments/<filename>` 图片 payload；Markdown 正文中的 `attachments/<filename>` 相对引用是真源，附件本身不是独立对象、不是 `.reo/index.json` 投影，也不进入 manifest 列表。
- Memory space registry entry 属于 main-owned app state，字段包含 `workspaceId`、title、description、rootPath、addedAt 和 lastOpenedAt；renderer 只能读取不含 rootPath 的列表投影。
- Voice transcription settings 属于 main-owned app state，文件位于 `userData/voice-transcription-settings.json`；X-Api-Key 字段经 Electron `safeStorage` 加密，renderer 只能读取不含明文或密文的 settings snapshot。同一 X-Api-Key 是录音中流式识别和 finalized audio 补转录的凭证真源。
- `workspaceHandle` 是 main process runtime capability，只用于当前窗口授权 IPC 操作，不进入 query key、DOM、URL、记忆空间文件或 registry。

## 技术方向

- 本地 SQLite 使用 Drizzle ORM + `better-sqlite3`。
- 没有真实 schema、关系和 migration owner 前，不引入 Drizzle、`better-sqlite3`、Drizzle config 或 migration directory。
- 每个功能变更都必须显式判断是否产生 DB schema、table relationship、query key、cache ownership、form owner、client state owner 或 durable file contract。
- 如果判断为不引入 DB schema，必须说明当前 durable source、恢复路径、未来触发 DB 的具体 pressure。
- Auth/session foundation 使用 Better Auth 的 Electron 支持。
- 没有真实 session lifecycle、auth tables 和 secure persistence owner 前，不引入 Better Auth package 或 auth storage。
- Form、IPC、auth、persistence 边界使用 Zod 做运行时校验。
- Main/server-backed async data 使用 TanStack Query。
- TanStack Query provider 当前服务真实 main-backed memory space list、记忆空间 snapshot consumer、Memory detail consumer、selected Segment content consumer 和 selected SegmentSupplement content consumer。Memory detail cache 是按 `workspaceId + memoryId` 标识的小型投影 cache；进入或离开同一 workspace session 时保留该投影用于即时渲染，并在新 session ready 后标记 stale，由新 handle 后台刷新。Segment content 和 SegmentSupplement content cache 持有本地音频 bytes，进入或离开 workspace session 时按 workspace 清理。
- 非 server-backed 的本地 UI/client state 使用 Zustand。
- 没有跨 component subtree 的 client state owner 前，不引入 Zustand store。
- Form state 使用 React Hook Form。
- React Hook Form 当前只服务真实 create memory space form submit、folder selection draft、Memory create title draft、Memory rename title draft、Segment rename title draft、SegmentSupplement rename title draft、memory space rename title draft 和 validation owner。

## Schema 规则

- 从产品实体和关系定义表，不从 UI screen 倒推表。
- 实现 migration 前必须写清每个 table relationship。
- 设计表时必须先写 entity、relationship、cardinality、ownership、lifecycle、delete/update effect，再写 columns。
- 优先显式 foreign key 和 index，不靠隐式耦合。
- 不为同一 domain concept 建多个重复 table shape。
- 不创建 speculative columns 或 generic metadata bucket。
- Migration 必须可审查，并关联引入它的 spec。

## 状态归属

- 用户记忆内容的 durable artifact source 属于记忆空间文件。
- `.reo` metadata 和 rebuildable index 属于记忆空间文件夹，由 Reo 管理。
- Memory space registry 属于 main-owned app state，只保存已导入记忆空间列表和 main-only root path，不是用户内容真源。
- SQLite 只有在引入后才拥有明确的 app index、relationship、query、session 或 processing state；不得替代记忆空间文件成为用户内容真源。
- Main/server-backed async data 属于 TanStack Query。
- Ephemeral UI state 属于 component state 或 Zustand。
- Form drafts 在提交前属于 React Hook Form。
- Derived state 默认计算，不存储；只有明确性能或一致性理由才存储。

## 当前数据决策

- 当前用户内容真源是记忆空间文件夹；不引入 Drizzle schema、SQLite file、migration directory 或 DB-backed content truth。
- Memory space registry 是 Reo app state，用于跨 app restart 保留已导入记忆空间。Registry entry 包含 `workspaceId`、title、description、root path、addedAt 和 lastOpenedAt；root path 只在 main-owned registry file 和 main process 内部存在。Renderer 只能通过 `workspace:listMemorySpaces` 读取不含 root path 的记忆空间元数据，通过 `workspace:openMemorySpace(workspaceId)` 打开记忆空间，并通过 `workspace:removeMemorySpace(workspaceId)` 从列表移除记忆空间。Registry 最多保留最近 100 个记忆空间，title/description 有 display text 长度上限；root path 超出上限时 registry write 失败。
- Voice transcription settings 由 main process `voiceSettingsStore` 持有，文件 `userData/voice-transcription-settings.json` 经 `safeStorage` 加密 X-Api-Key 字段；renderer 通过 TanStack Query `['settings', 'voice']` 读取不含明文或密文的 snapshot。Snapshot 字段为 `enabled`、`apiKeyConfigured`、`apiKeyLastFour`、`lastValidatedAt`、`lastValidationOk` 和 `lastValidationCode`。`setEnabled`、`saveApiKey` 和 `clearApiKey` mutation 成功后用 response snapshot 直接 seed exact key；`validate` mutation 成功后 invalidate exact key 以重新读取更新后的 validation snapshot。若 `saveApiKey` 已写入 key 文件但 validation snapshot 写入失败，main 返回 `file-written-index-stale`，renderer invalidate `['settings', 'voice']` 并清空本次草稿，因为 durable key 已由 main 持有且只能以 last4 投影展示。该 query 是 application-scoped，不绑定 `workspaceHandle`，不在 workspace session 切换时清理。
- Recording transcription session 在 start 时通过 `resolveVoiceSettings()` 拍下当前 voice settings 快照，并把 `transcriptionMode: 'live' | 'disabled'` 写入 `WorkspaceRecordingTranscriptionControlResponse.value`。Live session 一旦 start 完成就使用该次 start 的 settings 快照，不响应中途的 voice settings 变更。
- Backfill queue 在每个任务执行时从 `voiceSettingsStore` 读取当前解密后的 X-Api-Key，不把 key 复制进 queue entry、Query cache、renderer state、diagnostic payload 或记忆空间文件。Turbo request 的 `audio.data` base64 只在 main process 任务内存中存在，任务结束后不持久化。
- Memory space registry 不是记忆空间内容真源；记忆空间 root folder basename 是 title 文件真源，`.reo/workspace.json.title` 是 metadata mirror，description、memory summary 和 segment truth 仍来自记忆空间文件。Memory space title 不写入 `.reo/index.json`；`.reo/index.json` 只保存可重建的 `memories[]` summary 投影。Registry 文件缺失、损坏、schema 不匹配或 symlink leaf 按空记忆空间列表处理；不可读 IO 错误返回 typed error envelope。Registry upsert 在单 registry 实例内串行化，避免并发导入覆盖既有记忆空间。List 只读取 registry file，不扫描每个 root。Open 已导入记忆空间时只 resolve 当前 `workspaceId`：stored root 可读且 metadata `workspaceId` 匹配时使用该 root；stored root 不存在时，才在原父目录下流式检查直接子目录，最多读取 1000 个 sibling entry，并最多探测 200 个带合法 `.reo/workspace.json` 的 Reo workspace metadata candidate，查找同一 `workspaceId` 的 Reo 记忆空间 folder；不含 Reo metadata 的普通 sibling directory 不消耗该 candidate cap，但仍消耗总 sibling scan cap。Registry 读取和解析不写入记忆空间文件；Finder 改名后的 folder basename 只有在 open 获取 single-writer lock、且在任何 recovery/index 写入前只读确认打开的 `workspaceId` 匹配 registry entry 后，才只写回 `.reo/workspace.json.title` mirror 并 upsert registry projection。
- Memory space rename 使用 `workspace:updateMemorySpaceTitle`。Active 记忆空间 request 只包含 `workspaceHandle` 和安全 trimmed title；inactive registry entry request 只包含 `workspaceId` 和安全 trimmed title。Main process 在 single-writer lock 下把记忆空间 root folder basename 改成 title，并在 root move 成功后写入 `.reo/workspace.json.title` mirror；同名 sibling folder 已存在时返回 `ERR_WORKSPACE_ALREADY_EXISTS` 并保留当前 root、metadata、registry 和 `.reo/index.json`。Root move 是提交点；最终 root move 使用 no-overwrite platform move，并在 move 后用 directory identity 判定成功或冲突；case-only rename 在大小写不敏感文件系统上以目标 path 解析到原 root identity 且 platform move 成功作为成功；当前平台没有 no-overwrite directory move primitive 时返回更新失败，不冒险替换目标目录。Root move 和 metadata mirror 提交前不得读取或重建 `.reo/index.json`；返回成功 snapshot 所需的 index 投影只在 root move 与 metadata mirror 成功后读取或重建。Root move 成功后若 metadata mirror 或后续 projection 失败，错误必须使用 `file-written-index-stale`，不得声明旧文件已保留。Active path 以 root move 和 metadata mirror 成功作为保存成功，registry projection 写入失败不阻断当前会话 response；root move 成功后 active handle 和 lock 必须迁移到新的 canonical root，后续 snapshot read 和 close 继续使用同一 opaque handle。Inactive path 需要先通过 registry 解析 root，并在文件真源更新后更新 registry title/description/root projection，但保留该 entry 的 addedAt、lastOpenedAt 和列表顺序。Response 返回更新后的 Workspace snapshot，不返回 root path 或 handle。Renderer 提交后立即关闭 Dialog、更新当前 session snapshot 和 memory space list 投影；普通保存失败只在当前 title 仍是本次提交值时回滚，`file-written-index-stale` 失败保持 optimistic title 并用 root toast 显示错误。
- 用户直接删除记忆空间文件夹后，registry entry 可以继续显示在 sidebar。点击该记忆空间时 main 返回 `ERR_WORKSPACE_ROOT_MISSING`；renderer 通过 root toast 显示文件夹已不存在，不自动移除 registry entry，不切换 active session。用户可用 sidebar 记忆空间更多菜单把该 entry 从 Reo 列表移除；该动作不删除本地文件夹。
- Memory space stable files 包括 `AGENTS.md`、`.reo/workspace.json`、`.reo/index.json`、`.reo/objects/memories/<memoryId>.json`、`.reo/objects/segments/<segmentId>.json`、`.reo/objects/supplements/<supplementId>.json`、`memories/<memoryDirectory>/memory.md`、`memories/<memoryDirectory>/segments/<segmentDirectory>/segment.md`、audio Segment 的 `memories/<memoryDirectory>/segments/<segmentDirectory>/audio.webm`、`memories/<memoryDirectory>/segments/<segmentDirectory>/supplements/<supplementDirectory>/supplement.md`、audio SegmentSupplement 的 `memories/<memoryDirectory>/segments/<segmentDirectory>/supplements/<supplementDirectory>/audio.webm`、`.reo/trash/memories/<memoryDirectory>/` 下的可恢复已删除 Memory 目录、`.reo/trash/segments/<segmentDirectory>/` 下的可恢复已删除 Segment 目录，以及 `.reo/trash/supplements/<supplementDirectory>/` 下的可恢复已删除 SegmentSupplement 目录。Note Segment 和 note SegmentSupplement 使用对应 Markdown 正文与 `.reo/objects/*/*.json` 的 `bodyByteLength`，不包含 `audio.webm` payload；finalized note Segment 或 note SegmentSupplement 可按需包含同目录下的 `attachments/<filename>` 图片 payload。Draft 目录可包含 draft-local `segment.json`、`supplement.json`、`segment.md`、`supplement.md`、`audio.webm` 和 `transcript.md`。
- `memory.md`、`segment.md` 和 `supplement.md` 是用户和 agent 可编辑的语义真源；`.reo/workspace.json`、`.reo/objects/*/*.json`、draft、trash、lock 和 recovery marker 是 Reo 管理的技术完整性层。用户内容目录中的普通 `.json`、`.md`、`.html` 或其它文件可以存在，但不自动成为 Reo 对象，也不作为 Reo schema 输入。HTML 默认是不可信资源，未进入隔离预览能力前不由 renderer 执行或渲染。当前文件合同识别的 Markdown/frontmatter、`.reo/workspace.json`、`.reo/index.json` 和 `.reo/objects/*/*.json` 出现非法 frontmatter、非法 JSON、schema 不匹配、identity 不一致、symlink leaf 或 unsafe path 时返回 typed error 并保留既有文件，不自动覆盖成缓存状态。
- `.reo/index.json` 是可重建 UI index，不是用户内容真源。
- `.reo/index.json` 当前保存按 Memory 投影 `updatedAt` 倒序排列的 `memories[]`；每个 summary 包含 `memoryId`、title、createdAt、updatedAt、segmentCount、audioSegmentCount、noteSegmentCount、audioDurationMs、audioByteLength、hasAudioTranscript、hasAnyNote 和 supplementCount。`hasAnyNote` 表示存在 note Segment 或 note SegmentSupplement。Memory 投影 `updatedAt` 取自身内容时间和子孙 Segment/SegmentSupplement 内容时间的最大值；title rename 只改变可见名称和 Markdown title mirror，不改变 activity 排序时间。
- `.reo/workspace.lock` 和 `.reo/workspace.lock.lock` 是 volatile runtime lock artifacts，不进入 Codex read-only validation 的稳定 hash 范围；`.reo/workspace.lock.lock/owner.json` 记录当前 main process pid、进程启动指纹和 schema version，用于识别 stale lock，不是用户内容。缺失、损坏、symlinked、进程启动指纹不匹配，或不含启动指纹且 owner file mtime 早于当前 pid 启动时间的 owner file 代表未完成或无效 lock，可在下一次 acquire 时替换。
- Query keys 使用 stable `workspaceId`、`memoryId`、`segmentId` 和 `supplementId`；memory space list 使用 `['workspace', 'memory-spaces']`；Workspace snapshot 使用 `['workspace', 'snapshot', workspaceId]`；Memory detail 使用 `['workspace', 'memory-detail', workspaceId, memoryId]`；selected Segment content 使用 `['workspace', 'segment-content', workspaceId, memoryId, segmentId]`；selected SegmentSupplement content 使用 `['workspace', 'segment-supplement-content', workspaceId, memoryId, segmentId, supplementId]`；voice transcription settings 使用 `['settings', 'voice']`；`workspaceHandle` 是 main memory capability，不进入 query key、不写入文件、不跨 app restart 持久化。
- 手动补转录成功后复用现有 transcript save response 数据更新 Workspace snapshot、Memory detail、Segment content 和 SegmentSupplement content cache。手动补转录 running state 使用按 workspace/entity identity 建立的 App feature-local Map，并记录发起时的 `workspaceHandle`；进入新的 workspace session 时清理不属于当前 handle 的 running key，任务 settle 时只清理仍绑定原 handle 的 key。`fill-missing` / `regenerate` mode 只存在于 manual request 和 main queue task，不成为 Query key，不写入 Zustand store，也不写入持久字段。
- Query key 不包含 `workspaceHandle`。Renderer 在设置 ready workspace session 前，以及成功 close 当前 workspace 后，会移除同 `workspaceId` 下的 Segment content 和 SegmentSupplement content queries；Memory detail cache 保留，并在新 session ready 后用 `refetchType: "none"` 标记 stale，避免旧 handle observer 立即 refetch，同时让新 Memory Studio mount 时用新 handle 后台刷新。Workspace snapshot cache 由新的 ready snapshot 重新 seed。
- TanStack Query 拥有 main-backed memory space list、记忆空间 snapshot cache、Memory detail cache、selected Segment content cache 和 selected SegmentSupplement content cache。Selected note Segment content 和 selected note SegmentSupplement content cache 持有 `bodyMarkdown`、`bodyByteLength` 和当前正文的 `baselineContentHash`；baseline hash 只用于下一次 finalized note 正文保存的外部修改检测，不写入 Markdown、manifest、index、DOM 文案、toast 或日志。Segment delete 的 renderer grace period 使用 Workspace snapshot cache、Memory detail cache 和当前 session projection 做 scoped optimistic update；pending projection 以 - Finalize response 返回当前 Memory summary 和完整 finalized audio 或 note Segment projection；renderer 把 Memory summary 合并进当前记忆空间 session 的 `memories` 投影，把该 Memory 设为 current memory context，并把 Segment projection 合并到对应 Memory detail cache 的 `segments[]`。Transcript save 成功 response 返回当前 Memory summary；renderer 更新 Workspace snapshot 和 session summary，把对应 Memory detail cache 内的 Segment transcript presence 标记为存在，并把 `lastTranscriptionAttempt` patch 为 `'success'`，再 invalidate 该 Segment content Query；main 同时把该 Segment manifest 的 `lastTranscriptionAttempt` 写为 `'success'`。Note Segment body save 成功 response 返回 bodyByteLength 和新的 `baselineContentHash`；renderer patch exact Segment content Query 的 bodyMarkdown/bodyByteLength/baselineContentHash，并 patch Memory detail cache 中对应 note Segment 的 bodyByteLength。SegmentSupplement transcript save 成功 response 返回当前 Memory summary、parent Segment projection 和 supplement projection；renderer 更新 Workspace snapshot、session summary 和 Memory detail cache，并 invalidate 该 supplement 的 exact content Query；main 同时把该 SegmentSupplement manifest 的 `lastTranscriptionAttempt` 写为 `'success'`。Note SegmentSupplement body save 成功 response 返回 bodyByteLength 和新的 `baselineContentHash`；renderer patch exact SegmentSupplement content Query 的 bodyMarkdown/bodyByteLength/baselineContentHash，并 patch Memory detail cache 中 parent Segment 的目标 note supplement bodyByteLength。SegmentSupplement finalize response 返回当前 memory summary、parent segment projection 和 supplement projection；renderer 用 memory summary 更新 Workspace snapshot，用 parent segment projection 精准更新 Memory detail cache 的对应 segment `supplementCount` 和 `supplements[]`，但不把 supplement 作为同级 Segment 插入顶层 Segment strip。durable truth 仍是记忆空间文件和 `.reo/index.json` 的 memory summary。
- Note image attachment save response 不写 Query cache、不改变 Memory summary，也不改变 `.reo/index.json`。Renderer 只把 save response 的 relative `attachments/<filename>` 插入当前 markdown draft；正文保存成功后，Markdown 相对引用才成为正文语义真源。附件 payload 已写入但正文未保存时，文件可以作为同一 note 文件空间节点下的无引用 payload 留存；Reo 当前不实现附件 garbage collection、file watcher 或独立附件对象索引。
- Zod 当前用于 workspace IPC request/response、记忆空间 metadata、segment metadata 和错误信封。
- `chooseDirectory` 阶段不产生 durable data contract；真实路径只暂存在 main process selection token store，不写入文件、不进入 renderer、不进入 query key。
- Memory space initialize 在所选父目录下 no-replace 创建 title 同名 child folder，再在 child root 写入 `AGENTS.md`、`.reo/workspace.json`、`.reo/index.json`、`memories/` 和 `.reo/drafts/segments/`；同名 child 已存在时返回 `ERR_WORKSPACE_ALREADY_EXISTS`。`.reo/drafts/supplements/` 只在 SegmentSupplement 录音或笔记草稿创建时按需创建。open 现有记忆空间时会在返回 ready session 前补齐缺失的 `memories/` 和 `.reo/drafts/segments/` 托管目录；open 空文件夹时会在该文件夹内原地初始化 Reo 文件；open 非空非 Reo 文件夹返回 metadata invalid。initialize/open 获取 lock 后，托管目录创建、metadata revalidation、recovery、valid index read、必要 index rebuild 和 session 返回前都必须消费当前 lock/root/`.reo` usability；中途失去 ownership 时返回 lock lost，不能继续写入新的 `.reo` 或记忆空间文件。`AGENTS.md` 使用 no-replace atomic write，不能覆盖用户已有文件。
- `.reo` metadata directory、`.reo/drafts`、`.reo/drafts/segments` 和 `memories/` 必须是缺失或真实目录；如果任一层是 symlink 或非目录，initialize/open/lock/draft 初始化都必须拒绝，不能跟随 symlink 写入 记忆空间外。`.reo/workspace.json` 和 `.reo/index.json` 读取必须使用 no-follow file handle，symlink leaf 不作为 记忆空间 metadata 或 index 输入。托管 child directory 必须在已验证 parent identity 内相对创建，创建前后复核 parent，不能在 ancestor swap 后复用缓存 absolute path。Workspace lock 必须绑定当前 memory space root identity、`.reo` directory identity 和 lock directory identity，在当前 `.reo` directory identity 内打开 no-follow `workspace.lock` leaf 并创建同目录 `workspace.lock.lock`；owner file 必须在已绑定 lock directory identity 内 no-follow 创建、写入 main process pid 与进程启动指纹并 fsync，owner 进程已不存在、进程启动指纹不匹配或 owner file 无效的 stale lock 可被替换；不含启动指纹的 owner file 通过 owner file mtime 与当前 pid 启动时间判断 PID 复用。symlink leaf、root swap、parent swap 或 lock directory replacement 必须让 lock 不可用于后续 handle 授权。
- Audio segment draft 写入 `.reo/drafts/segments/<segmentId>/segment.json` 和 `audio.webm`；draft path 的 `.reo`、`.reo/drafts`、`.reo/drafts/segments` 和 leaf segment directory 任一层是 symlink 时必须拒绝写入。Draft create 在 parent identity 改变后必须清理自己刚创建的空 draft directory；当前进程创建的 active draft 在 append hot path 不重复扫描 finalized segments，但记忆空间 close、window teardown 或 root runtime state 清理后必须移除该进程态，后续 append 重新检查 finalized truth。Append metadata 写失败时必须将 audio 截回 append 前 size，返回 `ERR_RECORDING_APPEND_FAILED` 与 `dataRetention: "draft-preserved"`，且不得推进 sequence 或 byte count。cursor 大于 0 的替换时 target draft 必须先保持空 audio，再由 `workspace:cloneRecordingDraftPrefix` 从 source draft 复制 retained byte prefix；copy 前后必须校验 source/target metadata 和 target 空状态，复制中 lock lost 或 IO failure 必须回滚 target audio 为空，不得改变 source draft。cursor 为 0 的替换不克隆旧 draft，不保留旧 WebM 后段，必须创建新的 recording session 和新 draft 后重新写入 audio。Finalize request 必须带显式 `memoryId` 和 `durationMs`；loaded workspace 录音优先使用当前 Memory context，只有没有当前 Memory 时才先创建 Memory，main 只把 audio segment 追加到显式 Memory，不允许从 `segmentId` 隐式派生 Memory。Finalize 后只允许从 draft 读取 `segment.json`、`audio.webm` 和 `transcript.md`，并在 matching Memory 的 `segments/<segmentDirectory>/` 暴露 `segment.md`、`audio.webm` 与 `.reo/objects/segments/<segmentId>.json`；未知普通文件、symlink 或非文件必须让 finalize 失败并保留 draft。Durable Segment manifest 写入 `memoryId`、`segmentId`、`kind: "audio"`、duration、`finalizedAt`、updatedAt、`lastTranscriptionAttempt` 和实际 `audio.webm` 字节数，transcript 文本写入 `segment.md` 正文，并更新 `.reo/index.json`。
- SegmentSupplement recording finalize 在 parent Segment 的 `supplements/<supplementDirectory>/` 暴露 `supplement.md`、`audio.webm` 与 `.reo/objects/supplements/<supplementId>.json`；Durable SegmentSupplement manifest 写入 `workspaceId`、parent `memoryId`、parent `segmentId`、`supplementId`、`kind: "audio"`、duration、`finalizedAt`、updatedAt、`lastTranscriptionAttempt` 和实际 `audio.webm` 字节数，并刷新父 Memory summary。
- Finalize 在写 durable Segment manifest 和 `segment.md` 前必须把复制后的 draft metadata 作为不可信文件重新按 draft schema 校验，并确认 `workspaceId`、`segmentId`、`type: "audio"` 和 `audioByteLength` 与当前 finalize 请求及实际 `audio.webm` 一致；校验失败必须在暴露 durable segment truth 前失败并保留 draft。
- Finalize 创建 matching Memory 的 `segments/` 时必须逐段使用非递归目录创建和 `lstat` 复核；如果 `memories/`、matching Memory directory 或 `segments` 在目标解析后、实际 `mkdir` 前被替换成 symlink，必须拒绝 finalize、保留 draft，且不得在 记忆空间外创建 memory、segments 或 staging tree。Staging directory 必须在已验证 `segments/` parent 内用相对 `mkdir` 创建，并在 marker、fsync、copy 和 expose 前重新解析，不能复用 validation 前缓存的 absolute path；staging draft `segment.json` 必须通过 no-follow file handle 读取。最终 expose 必须在已验证 parent directory 内重验 target missing，创建 target directory 作为 no-replace reservation，先搬迁 marker 再搬迁 staging 内容；发现同名 target 已存在或 late-created target 必须失败并保留 draft，不得替换用户文件。
- Memory manifest 必须满足 `.reo/objects/memories/<memoryId>.json.memoryId` 与当前 lookup 请求匹配；`memory.md` 和对应 manifest leaf 必须用 no-follow file handle 读取，symlink leaf 不进入 rebuild/index/read 投影。
- Finalized audio segment 必须同时满足：`segment.md` frontmatter 合法、`.reo/objects/segments/<segmentId>.json.objectType === "segment"`、manifest `kind === "audio"`、manifest `memoryId` 与 parent Memory manifest id 一致、manifest `segmentId` 与 lookup 请求匹配、title、duration 和 audio byte 投影字段通过 schema 校验，且 segment leaf directory 不是 symlink，`segment.md` 与 manifest leaf 通过 no-follow file handle 读取，manifest `audioByteLength` 与 `audio.webm` 实际文件大小一致。finalized audio segment 不接受后续 audio append；即使存在同 id stale draft，append 也必须先识别 durable finalized truth 并返回 finalized error。
- Finalized note Segment manifest 必须同时满足 `.reo/objects/segments/<segmentId>.json.objectType === "segment"`、manifest `kind === "note"`、manifest `memoryId` 与 parent Memory manifest id 一致、manifest `segmentId` 与 lookup 请求匹配、body byte length 合法，且允许同一 Segment manifest 持有当前内容 tab 顺序字段 `contentTabOrder`。Note Segment 正文 read/write schema 必须把 `contentTabOrder` 作为当前已知字段接受并随 manifest update 保留；不得因为已持久化内容 tab 顺序而阻断 `segment.md` 正文读取、保存或内联编辑。
- Finalized audio segment transcript save public request 必须携带 `workspaceHandle`、`memoryId`、`segmentId` 和 markdown；用户编辑、清空或 guarded overwrite 会额外携带 `baselineTranscriptHash`。Main process 按 manifest id 定位 matching Memory 与 matching Segment 的文件真源，不通过全局 `segmentId` lookup 选择 first match。已有重复 finalized `segmentId`、unsafe finalized Markdown/manifest 或 manifest ownership mismatch 必须返回 typed error envelope。携带 baseline 时，main 必须先比较当前 `segment.md` transcript digest；不匹配时返回 `ERR_BACKFILL_TRANSCRIPT_CHANGED`，不执行 Markdown 覆盖、index refresh 或 manifest update。Transcript save 必须在当前 segment directory identity 内写入 `segment.md` 正文，不能在 parent swap 后使用缓存 absolute path。Save 成功 response 返回刷新后的 Memory summary 和新 transcript baseline hash，renderer 以该 summary 更新 Workspace snapshot；main 在写入 Segment manifest `lastTranscriptionAttempt: 'success'` 前必须复核 manifest 的 workspace、memory 与 segment ownership，保存失败或 ownership 变化时保留既有 manifest attempt。Finalized audio SegmentSupplement transcript save public request 必须携带 `workspaceHandle`、`workspaceId`、parent `memoryId`、parent `segmentId`、`supplementId` 和 markdown，并可携带 `baselineTranscriptHash` 做同一 transcript digest guard；main process 按 parent Segment 下的 supplement manifest id 定位真实 supplement 文件空间节点，写入该 supplement 的 `supplement.md` 正文，刷新父 Memory summary；main 在写入 SegmentSupplement manifest `lastTranscriptionAttempt: 'success'` 前必须复核 manifest 的 workspace、memory、parent Segment 与 supplement ownership，并返回父 Memory summary、parent Segment projection、supplement projection 和新 transcript baseline hash；该 response 不返回 transcript text、raw path、file path 或 handle；保存失败或 ownership 变化时保留既有 manifest attempt。Draft markdown save 只服务 in-progress draft transaction，不是 finalized public fallback。
- 如果 `.reo/index.json` 更新失败，finalize 返回错误信封并保留 draft metadata；不得留下 metadata finalized 但 index 缺失 summary 的不一致状态。
- 同一 `memoryId` 一次只允许一个 memory file write；create、append 和 title update 共用 main process memory write lock。并发 append 或并发 title update/append 会拒绝其中一个并保留未完成写入的既有文件状态，避免 last-writer-wins 丢失 recording。
- Workspace metadata、workspace index、`.reo/objects/*` manifests 都是受限 JSON 文件；`memory.md`、`segment.md` 和 `supplement.md` 是受限 Markdown/frontmatter 文件；读取必须 no-follow、普通文件、当前目录 identity 未变化，大小不得超过当前文件大小上限，schema 必须 strict，不得通过 unknown 字段偷偷扩展合同。
- Finalized note attachment payload 只允许写入对应 note Segment 或 note SegmentSupplement 目录下的 `attachments/` 一层子目录。`attachments/` 必须在已验证 owner directory identity 内相对创建或打开；读写和 list 都必须拒绝 symlink、非目录、非普通文件、parent swap 和路径穿越。附件文件名由内容 digest 前缀和安全 basename 生成；相同文件名已存在时只有 byte 完全一致才视为幂等成功，不覆盖不同内容。单个附件上限为 25 MiB，支持 MIME 只有 PNG、JPEG、WebP 和 GIF。
- `segmentId` 在所有 `memories/*/segments/*` 下必须全局唯一；已有重复 finalized `segmentId` 会让 lookup 拒绝返回 first match，避免不确定读写。
- 新建 Memory 容器不得复用已存在的同 id Memory 目录；重复 `memoryId` 或重复 durable `segmentId` 必须失败并保留既有文件，不得覆盖或删除用户内容。
- `.reo/index.json` 是可重建 index。Open 先读取合法 `.reo/index.json` 作为启动 cache，并用该 cache 返回 ready snapshot；只有 `.reo/index.json` 丢失、损坏或 symlink leaf 时才在 open 路径从文件真源 rebuild。Active renderer 在 session ready 后立即通过 `workspace:readWorkspaceSnapshot` 执行一次静默文件真源刷新，之后只在 document visibility 变为 visible 时刷新；普通 focus 不触发 refresh。Snapshot refresh 在 rebuild read model 或写 `.reo/index.json` 前，先按 root folder basename 修复 `.reo/workspace.json.title` mirror，避免 stale mirror 回灌 UI。Index rebuild 从 `memories/*/memory.md`、`memories/*/segments/*/segment.md`、`supplements/*/supplement.md`、`.reo/objects/*/*.json` 和对应文件 payload 协调并写回 summary，只纳入 schema、manifest ownership、finalized file-space ownership、audio 字节数一致的 finalized audio Segment / SegmentSupplement，以及正文 byte 长度一致的 finalized note Segment / SegmentSupplement。Transcript presence 和 content Query 的 `transcript.exists` 只存在于 audio projection，由 `segment.md` 或 `supplement.md` 的 `## Transcript` section 经 `extractSegmentTranscript` 提取后的 text 是否非空投影；普通 Markdown body text 不计入 transcript；symlink leaf 不进入 index。`memories/` scan 必须绑定 root directory identity；scan 前、scan 后或 index 持久化前 identity 改变时必须失败并保留既有 index。`memories/` 存在但不可读时，需要 scan 的 open rebuild 或 snapshot refresh 必须失败并保留既有 index，不能把 read error 写成空投影。Memory space title 不属于 index；memory space rename 失败于 root move 前时不得为了返回错误而协调 index。Workspace、Memory 和 Segment title update 在合法 index 存在时不扫描 Memory 文件树；Transcript save 和 SegmentSupplement finalize 只刷新所属 memory 的 index entry，不因单次保存重扫全 workspace。同一 workspace 的全量 index replace 和单条 index entry refresh 共用 main process index write queue。全量 replace 在队列内重新读取当前 read model；单条 refresh 每次在队列内部重新读取当前 memory summary，再合并当前 index，避免并发 metadata refresh 或 full rebuild 造成 stale summary 与 `.reo/index.json` last-writer-wins。
- Active session 的 Workspace snapshot 可以通过 `workspace:readWorkspaceSnapshot` 从文件真源重新读取。该 read 会读取当前 `.reo/workspace.json`，并通过 index rebuild/read model 协调外部修改后的 `memory.md`、`segment.md`、`supplement.md` 和 `.reo/objects/*` manifests；response 仍只返回不含 root path 或 handle 的 Workspace snapshot。
- 如果进程在 finalized metadata 写入后、`.reo/index.json` 写入前崩溃，下一次 active snapshot refresh 会按记忆空间文件真源协调合法但陈旧的 index；如果 index 缺失、损坏或 unsafe，open 会 rebuild。合法但陈旧的 index 只能作为启动 cache，不能让 finalized audio / note Segment 或 SegmentSupplement 永久隐藏。
- `updateWorkspaceIndex` 不在 update 成功前持久化 index 协调结果；如果 update 失败，已有合法 index 保持原状，避免 index summary 与回滚后的 draft metadata 反向不一致。

## 数据流设计纪律

- 每个 feature 必须先写 conceptual model，再决定是否需要 physical DB schema。
- 数据流必须写清 write owner、read owner、durable source、cache owner、mutation owner、invalidation 和 recovery。
- Workspace file、`.reo` metadata、rebuildable index、TanStack Query cache、component state 和 form state 不能存储同一事实的不同版本；Segment 关系真源只在 matching Memory 目录下的合法 finalized Segment 文件空间节点中出现，derived Workspace snapshot summary 是可重建投影。
- 如果同一事实需要出现在多个位置，必须说明 source of truth、同步方向、冲突处理和丢失恢复。
- 不得因为未来可能查询就提前建表；也不得因为第一版小就省略实体、关系和字段语义设计。
- Durable file write 必须定义 atomic rename 边界、file fsync、parent directory fsync、跨平台差异、失败时 `.part` 状态和恢复策略。
- Memory space write 必须定义 single-writer 策略；同一 workspace 被多个 Reo window 或进程打开时，必须设计检测、拒绝或合并语义。

## Query 规则

- Query keys 引入时必须稳定、命名清楚，并在本文档记录。
- 改变底层数据的 mutation 必须定义 invalidation。
- Optimistic update 必须在 `flow.md` 写 rollback。
- 不得为 main/server-backed data 绕过 TanStack Query 写临时 fetching。
- Zod 只用于不可信 runtime boundary，不用于重复 TypeScript 类型。

## Slice 审查门禁

每个 spec 在进入实现前必须回答：

- 是否需要 DB schema；如果需要，实体和表关系是什么。
- 是否需要 migration；如果需要，foreign key、index、delete/update effect 是什么。
- durable source 是 workspace file、`.reo` metadata、SQLite，还是多者分工；谁是真源。
- 数据获取模式是什么：TanStack Query、direct component state、filesystem scan 或其他明确 owner。
- Query keys、mutation invalidation、optimistic update 和 rollback 是否定义。
- Form state、client state、server/main-backed state 的 owner 是否清楚。
- 文件夹结构是否有 source-of-truth、rebuildable index、用户文件和 Reo metadata 的边界。

## 变更门禁

任何 schema、表关系、auth tables、migrations、query keys、cache ownership、Zustand ownership 改动，都必须更新本文档。
