# 流程

本文档是行为时序、生命周期和恢复策略的当前真源。

## 当前事实

- 当前 IPC request flows 覆盖 `window.reoWorkspace` 的 choose、list memory space entries、initialize、open、open memory space entry、remove memory space entry、close、Memory create、Memory detail read、finalized audio segment read、finalized audio SegmentAttachment read、memory title update、Memory delete/restore、recording draft、SegmentAttachment recording draft、transcript save、microphone intent begin/clear 和 recording transcription start/send/finish/close。
- 当前 preload/IPC consumer 覆盖记忆空间文件事务基础、create memory space UI flow、Memory create UI flow、MediaRecorder recording overlay 和 transcript save。
- 当前没有 auth/session lifecycle。
- 当前没有 auth request、exchange、sign-out 或 user-update flow。
- 当前记忆空间 file write 使用 atomic temp file、file fsync、rename 或 no-replace hard link、parent directory fsync 边界；可替换 metadata 用 rename，不能覆盖用户文件的 stable path 用 no-replace hard link。
- 当前没有 DB migration 或 startup database lifecycle。
- 当前没有 background queue。
- 当前 Query flow 覆盖 app 启动读取 memory space list、记忆空间 initialization/open 后 invalidate memory space list、memory space ready 后 seed 记忆空间 snapshot cache、选中 Memory 后读取 Memory detail cache、选中 Segment 后读取 Segment content cache、切换到 selected Segment 的 `补充` tab 后读取 SegmentAttachment audio content cache；snapshot cache 只承载记忆空间身份和 `memories[]` summary 投影；Memory detail cache 只承载当前 Memory 的 finalized audio Segment projection 和每个 Segment 的 finalized attachments projection；Segment content cache 只承载 selected finalized audio Segment 的 audio bytes 和 transcript 文本；SegmentAttachment audio content cache 只承载 selected Segment 下单个 finalized audio attachment 的 audio bytes；没有 optimistic update。
- 当前 App route flow 使用 renderer in-memory state：无 active memory space 时显示 starter Home shell，创建入口在 sidebar 记忆空间列表；`创建本地记忆空间` 打开受控 `WorkspaceCreateDialog` 弹层；memory space ready 后显示 loaded workspace frame；loaded workspace frame 的右侧 MemoryRail 只切换当前 Memory context，选中 Memory 后中间区域渲染 Memory Studio 并通过 Memory detail Query 读取当前 Memory 的 finalized audio Segment projection，Segment 横向预览流、时间轴和内容区只在当前 Memory 内同步 selected Segment；右侧 MemoryRail 默认折叠并退出可访问树，用户手动展开后按当前 workspace 宽度选择 layout mode：宽视口使用 inline 模式并让中央舞台和 FAB 使用同一 rail inset 避让，compact workspace 宽度使用 overlay 模式并覆盖在舞台上方，不改变中央舞台和 FAB 的右侧 inset；sidebar `首页` 或 `资料库` 在 active memory space 存在时先调用 `closeWorkspace` 释放当前 handle，再进入对应顶层页面；loaded workspace frame titlebar `新建记忆` 先命名 Memory 并保持 loaded workspace frame；loaded workspace frame 的 `录音` 优先使用当前 Memory context 作为 existing-memory recording target，只有没有当前 Memory 时才先命名 Memory。录音流程打开后，App 层会阻止顶层导航、创建或打开其他记忆空间、移除记忆空间和切换当前 Memory context；被阻止的 action 不得调用 open/close/remove workspace IPC，也不得清空当前 recording target。Finalize 后 App seed 记忆空间 snapshot cache 的 `memories[]` summary，把受影响 Memory 设为当前 Memory context，并把完整 finalized audio Segment projection 精准 upsert 到对应 Memory detail cache；SegmentAttachment finalize 后 App seed Workspace snapshot summary，并精准更新当前 Memory detail cache 的 parent Segment projection；recording transcript save 成功 response 返回当前 Memory summary，App 以该 summary 更新 Workspace snapshot cache 和当前 session state；该 route state 不写入 URL、记忆空间文件、Query key 或持久化 store。
- 当前 memory create flow 是 request/response mutation：Workspace titlebar `新建记忆` 或空 Memory 状态下的 `录音` 打开 `MemoryCreateDialog`，React Hook Form 持有提交前 title draft；renderer 调用 `workspace:createMemory`，main 通过 workspace handle 校验 sender 和 lock usability 后进入同 `memoryId` write lock，创建 `memories/<memoryId>/memory.json` 并刷新可重建 index，返回 Memory summary。Renderer 只在成功 response 后合并当前 session snapshot、seed snapshot cache，并把新 Memory 设为当前 Memory context；失败时 Dialog 保持打开并使用 root toast 显示可恢复错误，不写 optimistic state。
- 当前 memory rename flow 是 request/response mutation：MemoryRail More 菜单打开 `MemoryRenameDialog`，React Hook Form 持有提交前 title draft；renderer 调用 `workspace:updateMemoryTitle`，main 通过 workspace handle 校验 sender 和 lock usability 后进入同 `memoryId` write lock，读取并更新 `memory.json` 的 title/updatedAt，刷新可重建 index，并返回 Memory summary。Renderer 只在成功 response 后合并当前 session snapshot 和 seed snapshot cache；失败时 Dialog 保持打开并使用 root toast 显示可恢复错误，不写 optimistic state。
- 当前 memory delete/restore flow 是 request/response mutation：MemoryRail More 菜单打开 `MemoryDeleteDialog`，确认前不触发文件写入；确认后 renderer 调用 `workspace:deleteMemory`，main 通过 workspace handle 校验 sender、request schema 和 lock usability 后，把 active `memories/<memoryId>/` 移入 `.reo/trash/memories/<memoryId>/`，刷新 `.reo/index.json`，返回刷新后的 `memories[]` 和 `restoreToken`。Renderer 只在成功 response 后合并当前 session snapshot 和 snapshot Query cache、移除被删除 Memory 的 detail cache，并在 toast action 上保留本次 `restoreToken`；如果当前 Memory 被删，renderer 选择剩余第一条 Memory 或回到 Workspace Stage。用户点击 toast 恢复时 renderer 调用 `workspace:restoreDeletedMemory`，main 把 trash Memory 移回 active `memories/` 并刷新 index，renderer 合并 snapshot cache、选中恢复 Memory 并显示恢复成功 toast。删除或恢复失败只显示 root toast，不写 optimistic state，不清空原 projection。
- 当前没有 optimistic update path。
- 当前 memory space entry flow 覆盖 create 记忆空间 title/description/folder selection validation、创建时在所选父目录下新建同名记忆空间文件夹、sidebar 持续显示已导入记忆空间、sidebar `打开本地记忆空间` 打开现有 Reo 记忆空间或初始化空文件夹、点击已导入记忆空间打开该记忆空间、sidebar 记忆空间更多菜单移除 registry entry、open failure 可见错误反馈、submit failure 保留 create draft 和成功后 route state 切换。
- 当前没有 persisted client-state migration flow。
- 当前 main process 有最小 fatal exception path：`uncaughtException` 写入 `console.error`，先释放当前记忆空间 handles，再退出。
- 当前没有 structured diagnostic lifecycle 或 background error reporting flow。
- 当前没有 package、make、publish、release 或 update lifecycle。

## 技术方向

- 多步骤行为先建模，再实现。
- Command/event 名称必须具体、贴近 domain。
- 在没有真实 queue/stream 需求前，优先 request/response。
- 新 preload/IPC flow 只能跟随真实 renderer-to-main 特权需求创建。
- 在没有真实 schema 和 migration owner 前，不创建 DB migration flow。
- 每个 lifecycle transition 都必须有 owner。
- failure、cancellation、retry、recovery 行为必须明确。

## 状态管理纪律

- 状态机必须先描述 domain lifecycle，再决定使用 component state、reducer、React Hook Form、TanStack Query 或 Zustand。
- 不得把同一状态同时放入 component state、cache、form draft 和 metadata。
- UI 状态、form 状态、main-backed async data、durable file state 和 derived state 必须分开定义 owner。
- 复杂状态优先用 feature-local reducer 表达转移；只有跨 component subtree 且有真实 owner 时才引入 Zustand。
- 每个 state transition 必须说明触发事件、允许来源、失败行为、用户可见状态和验证路径。
- State ownership 必须用矩阵表达；每个 state row 只能有一个 source-of-truth owner，重复投影必须写同步方向和冲突处理。

## 当前流程决策

- Memory space directory selection flow：renderer 调用 `window.reoWorkspace.chooseDirectory()`；preload 调用 `workspace:chooseDirectory`；main 校验 channel、main frame、trusted URL 和 session；OS dialog canceled 返回 canceled；selected 返回 `selectionToken` 和 `displayPath`；真实路径只保存在 main 的 selection token store。
- Selection token lifecycle：issued、consumed、expired、sender-mismatch、not-found；成功 consume 和 expired 会删除 token，错误 sender 不删除 token；所有错误都不返回 path。
- Memory space initialize flow：renderer 传 selection token、title、description；main 校验 sender、token 和安全 title，把 token 指向的目录作为父目录，在父目录下 no-replace 创建同名 child memory space root；child 已存在返回 `ERR_WORKSPACE_ALREADY_EXISTS` 且不写入记忆空间文件。child 创建成功后获取 single-writer lock 并初始化 Reo 文件。lock file 写入前和打开前必须复核 `.reo` directory identity，lock target leaf 必须用 no-follow 打开并拒绝 symlink，lock directory 必须在同一 `.reo` 当前目录内创建并绑定 directory identity；owner file 必须在 lock directory 内 no-follow 创建、写入 main process pid 与进程启动指纹并 fsync。owner 进程已不存在、进程启动指纹不匹配，或 owner file 缺失、损坏、symlinked 的 stale lock 可被替换；无法读取进程启动时间且 owner pid 仍存活时返回 locked error。`AGENTS.md` 通过 no-replace atomic write 创建；写入记忆空间文件 后返回 opaque `workspaceHandle`、`workspaceId` 和 snapshot。
- Memory space open flow：renderer 传 selection token；main 校验 sender 和 token，canonicalize selected root；如果该目录是现有 Reo 记忆空间，则校验 metadata 和 unsafe children 后获取 single-writer lock 并打开；如果该目录为空，则获取 lock 后再次确认 root 除 lock-only `.reo` artifacts 外仍为空，再原地初始化为 Reo 记忆空间，title 使用 selected root basename，description 为空；如果第二次确认发现目录已变成非空非 Reo，返回 metadata invalid 并清理 lock-only `.reo`。非空非 Reo 目录、unsafe `.reo`、unsafe `.reo/drafts/segments`、unsafe `memories/` 或 invalid 记忆空间 metadata 返回错误且不得留下 lock。lock file 写入前和打开前必须复核 `.reo` directory identity，lock target leaf 必须用 no-follow 打开并拒绝 symlink，lock directory 必须在同一 `.reo` 当前目录内创建并绑定 directory identity；owner file 必须在 lock directory 内 no-follow 创建、写入 main process pid 与进程启动指纹并 fsync。owner 进程已不存在、进程启动指纹不匹配，或 owner file 缺失、损坏、symlinked 的 stale lock 可被替换；无法读取进程启动时间且 owner pid 仍存活时返回 locked error。corrupt index 可重建，corrupt metadata 阻断写入且不得留下 lock。
- Memory space registry flow：initialize/open 在记忆空间文件成功返回 snapshot 后、handle 注册前，把 canonical root 与 snapshot 写入 main-owned registry；registry upsert 失败时释放记忆空间 lock 并返回 `ERR_WORKSPACE_MEMORY_SPACE_REGISTRY_WRITE_FAILED`。Registry upsert 按单 registry 实例串行执行，避免并发导入互相覆盖。Registry 写入使用 atomic JSON write；读取使用 no-follow file handle、大小上限和 schema 校验。
- Memory space list flow：renderer 通过 TanStack Query 调用 `workspace:listMemorySpaces`；main 校验 sender、channel 和空 payload 后返回不含 root path 的记忆空间元数据。Registry 文件缺失、损坏、schema 不匹配或 symlink leaf 返回空记忆空间列表；不可读 IO 错误返回 `ERR_WORKSPACE_MEMORY_SPACE_REGISTRY_READ_FAILED`，renderer 使用 root toast 显示错误。Registry 写入最多保留最近 100 个记忆空间，title/description display text 有上限；registry 只是记忆空间列表投影，不是记忆空间文件真源。
- Memory space registry open flow：renderer 点击 sidebar 记忆空间时只发送 `workspaceId` 到 `workspace:openMemorySpace`。main 从 registry 解析真实 root；未找到返回 `ERR_WORKSPACE_MEMORY_SPACE_NOT_FOUND`；root folder 已被用户删除时返回 `ERR_WORKSPACE_ROOT_MISSING`；找到后复用 open target validation、single-writer lock、记忆空间文件 recovery/index reconciliation、registry upsert 和 handle registration，返回新的 opaque `workspaceHandle`。该 flow 不使用 selection token，也不向 renderer 暴露 root path。
- Memory space registry remove flow：renderer 在 sidebar 记忆空间更多菜单中确认“移除记忆空间”后，只发送 `workspaceId` 到 `workspace:removeMemorySpace`。main 只从 registry 删除 entry，不解析、不打开、不删除本地记忆空间文件夹；缺失 entry 也按移除完成处理。移除 active memory space entry 时，registry removal 是主路径；renderer 在移除成功后回到 starter Home，并尽力调用 `closeWorkspace` 释放当前 handle。close 失败不得恢复 registry entry 或阻断列表移除，只通过 toast 显示当前会话释放失败。
- Memory space initialize/open 在获取 lock 后，如果记忆空间文件 write、recovery、index rebuild、handle registration 或 response parse 抛出异常，必须释放 lock 并返回 workspace error envelope；lock target 写入失败必须返回 `ERR_WORKSPACE_LOCK_FAILED`，不得让 IPC handler reject。Lock 可用性同时要求 root identity、`.reo` identity 和当前 lock directory identity 未变化；initialize/open 获取 lock 后、写入或打开记忆空间文件前必须再次检查 lock usability；initialize 文件阶段在补齐托管目录和写入 `AGENTS.md`、记忆空间 metadata、index 前都必须重新消费 lock usability；open 文件阶段在 target revalidation、metadata read、补齐托管目录、recovery、read model rebuild、index reconciliation 和返回 session 前也必须重新消费 lock usability；identity 变化或 release 失败后，handle require 和 delayed handler operation 必须返回 lock lost，不得继续授权写入。
- Memory space close flow：renderer 传 `workspaceHandle`；main 校验 sender ownership，释放 lock 并撤销 handle。Handle close 只有在 lock release 成功后才能删除 handle；release 失败时保留 handle 并返回 `ERR_WORKSPACE_LOCK_FAILED`。BrowserWindow `closed`、`render-process-gone`、trusted main-frame non-same-document navigation 和 trusted main-frame reload 都会尝试释放当前 main process 持有的记忆空间 handles；same-document navigation、subframe navigation 和 untrusted navigation 不释放 handle。Navigation 和 window teardown 的 close-all 通过同一 lifecycle helper 合并重入释放；teardown close-all 只删除 release 成功的 handle，release 失败的 handle 保留为 lock lost 状态以便后续 close-all 重试。
- Memory space lifecycle 覆盖 none、creating、ready、missing、conflict、unsupported、failed。
- Memory space entry lifecycle 覆盖 idle、menu-open、create-dialog-open、folder-selecting、canceled、creating、open-local-memory-space-selecting、create-error、open-local-memory-space-error 和 ready。
- Memory space session replacement flow：同一个 renderer 窗口只显示一个 active 记忆空间 session。Create/open-local/open-memory-space 成功拿到新 `workspaceHandle` 后，如果已有 active memory space，renderer 必须先调用 `closeWorkspace` 释放既有 handle；既有 handle 释放失败时保留当前记忆空间 UI、使用 root toast 显示错误，并尽力关闭新 handle，避免双 lock 泄漏。打开 promise reject 时必须使用 root toast 显示可见错误。
- Memory space creation renderer flow：`创建本地记忆空间` 打开 `WorkspaceCreateDialog`；title、description、selection token 和 displayPath 属于当前 React Hook Form lifecycle；所选目录是 parent location，main 在 parent 下创建 title 同名记忆空间文件夹；OS dialog canceled 不修改 draft 并把 focus 返回 folder picker；同名 child 记忆空间文件夹已存在时使用 root toast 显示创建失败，保留 title/description，清除已消费的 selection token 和 displayPath；initialize success seed 记忆空间 snapshot Query cache、关闭创建弹层并设置当前 renderer session state。
- Memory space open-local renderer flow：sidebar `打开本地记忆空间` 单独调用 folder picker；OS dialog canceled 不改变 route state；selected folder 可以是现有 Reo 记忆空间或空文件夹，空文件夹会原地初始化为 Reo 记忆空间；renderer 只发送 selection token 给 `openWorkspace`，不发送 title、description、displayPath 或 raw path；open failure 使用 root toast 显示，不打开创建弹层、不清空当前记忆空间 session；open success 先按 记忆空间 session replacement flow 释放既有 handle，再 seed 记忆空间 snapshot Query cache、invalidate memory space list 并设置当前 renderer session state。该动作不进入 `WorkspaceCreateDialog`，也不路由到 `initializeWorkspace`。
- Memory create renderer flow：Workspace titlebar 的 `新建记忆` 打开 `MemoryCreateDialog`；title draft 属于当前 React Hook Form lifecycle；submit 只发送 `workspaceHandle` 和 trimmed title 到 `workspace:createMemory`。成功 response 合并当前 session snapshot、seed snapshot Query cache、把新 Memory 设为当前 Memory context、关闭弹层并显示该 Memory 的 Studio 基础态。创建失败使用 root toast 显示可见错误，不写 optimistic Memory card。
- Memory delete renderer flow：MemoryRail More 菜单的 `删除记忆` 打开 `MemoryDeleteDialog`；Dialog open state 属于 App component state。确认按钮 pending 时禁止重复提交；取消或关闭不改变 Workspace snapshot。删除成功 response 是唯一 UI 更新来源，renderer 用 immutable update 替换当前 `memories[]` 投影、移除被删 Memory 的 detail cache，并根据当前 selection 决定切换到剩余第一条 Memory 或 Workspace Stage。toast action 的 `恢复` 调用 restore flow；toast 之外当前不提供 trash browser。
- Memory delete/restore main flow：`workspace:deleteMemory` 和 `workspace:restoreDeletedMemory` 都必须在当前 handle owner、workspace root identity、`.reo` identity 和 lock usability 有效时执行。Delete move、restore move、index refresh 和 rollback 都只能在已验证 workspace child path 内进行；`.reo/trash/memories/` 缺失时按需创建，任何 symlink、非目录、ancestor swap 或 active/trash directory conflict 都返回 typed error envelope。Delete 在 move 成功但 index refresh 失败时尝试移回 active memory directory；restore 在 move 成功但 index refresh 失败时尝试移回 trash directory。Rollback 失败时返回错误并保留文件真源，不向 renderer 报告已完成删除或恢复。
- Memory detail renderer flow：选中 Memory 后，Memory Studio 使用 `['workspace', 'memory-detail', workspaceId, memoryId]` Query key 调用 `workspace:readMemoryDetail`。Request 携带 `workspaceHandle`、`workspaceId`、`memoryId` 和 requestId；main 校验 handle owner、workspaceId 与 handle workspace 匹配、lock usability 和 Memory 文件真源后，返回同 requestId 的 Memory detail projection。Renderer 只接受 requestId、workspaceId 和 memoryId 与当前 Query 匹配的 response；不匹配视为 stale response，不写当前 UI。
- Segment selection renderer flow：Memory detail loaded 且存在 Segment 时，Memory Studio 默认选择当前 detail projection 的首个 Segment。普通录音 finalize 后，App 把完整 Segment projection 合并到当前 Memory detail cache，并传入一次性 Segment focus intent；Memory Studio 只在该 Segment 已进入当前 detail projection 后消费 intent 并选中该 Segment。用户点击 Segment 横向预览 item 时，只更新 feature-local `selectedSegmentId`，并同步当前 card、时间轴点和内容区；该 item 是唯一交互目标，内含录音卡片、timeline 圆点和时间标签，不再暴露独立 timeline button。该选择不写入文件、URL、Query key、Workspace snapshot 或 registry。切换 Memory 会重置 Memory Studio component state，避免旧 Segment selection 留到新的 Memory。
- Segment strip carousel renderer flow：Memory Studio 的 Segment 横向预览流通过 feature-local DOM ref 读取 `scrollWidth`、`clientWidth` 和 `scrollLeft`，在初始化、scroll、ResizeObserver 和 window resize 后重新计算左右 `CarouselArrowButton` 显示状态。Segment item 是该 scroll owner 的直接子项，item 宽度使用紧凑 `--memory-studio-segment-card-size`，card、圆点和时间标签共同随同一横向滚动位移。不可达方向不渲染按钮，因此不可点击也不可被键盘聚焦。点击左右按钮只调用该横向预览流的平滑滚动，不更新 `selectedSegmentId`、Query cache、Workspace snapshot 或 durable file。横向溢出只属于 Segment strip scroll owner，不改变 document scroll，也不撑开 WorkspaceFrame。
- Finalized audio Segment content renderer flow：Memory detail loaded 且 selected Segment 存在时，Memory Studio 使用 `['workspace', 'segment-content', workspaceId, memoryId, segmentId]` Query key 调用 `workspace:readFinalizedAudioSegment`。Renderer 只接受 requestId、workspaceId、memoryId 和 segmentId 与当前 Query 匹配的 response；不匹配视为 stale response。Query 成功后，Memory Studio 用 audio bytes 创建 local Blob URL，`播放片段` 调用 hidden HTMLAudioElement `play()`，`暂停片段` 调用 `pause()`，ended、Segment 切换、content 变化和 unmount 会停止播放并释放 Blob URL。Transcript 只显示 `transcript.md` 的真实文本；缺失或空 transcript 显示空态，不生成 mock 文本。
- Finalized audio SegmentAttachment content renderer flow：Memory detail loaded、selected Segment 存在且 selected Segment 有 finalized attachments 时，Memory Studio 才显示 `补充` tab。用户切换到 `补充` tab 后，每个 audio SegmentAttachment 使用 `['workspace', 'segment-attachment-content', workspaceId, memoryId, segmentId, attachmentId]` Query key 调用 `workspace:readFinalizedAudioSegmentAttachment`。Renderer 只接受 requestId、workspaceId、memoryId、segmentId 和 attachmentId 与当前 Query 匹配的 response；不匹配视为 stale response。Query 成功后，attachment player 用 audio bytes 创建 local Blob URL，播放/暂停/seek 与 selected Segment playback 使用同一 Waveform UI 语义，但不读取或展示 transcript 文本。读取失败时在该 attachment 行显示 `补充录音加载失败。`，不展示 transcript 占位。Attachment player unmount 或 content 变化时释放 Blob URL。
- Segment content tab and attachment menu renderer flow：Memory Studio 的内容 tab 属于 selected Segment 的 feature-local UI state；audio Segment 始终显示 `转录` tab，只有 selected Segment 存在 finalized attachments 时才显示 `补充` tab。同一 selected Segment 的 finalized attachments 从 0 变为大于 0 时，active tab 自动切到 `补充`；切换 Segment 或切换 Memory 时 active tab 回到 `转录`。`笔记`、`视频` 和 `图片` 不作为常驻禁用 tab 渲染，也不触发 IPC、Query、文件写入或 mock 内容。内容 tab 行右侧的 `+` 是 selected Segment 的 SegmentAttachment 菜单入口，不属于 tab 本身；菜单 open 状态只存在于 `MemoryStudio` component state，切换 Segment 或切换 Memory 时关闭。当前菜单只暴露录音补充项；选择后打开 `segment-attachment` recording target，显式携带 parent `memoryId` 和 `segmentId`。
- SegmentAttachment recording renderer flow：录音 overlay 在 `segment-attachment` target 下复用同一 recording surface 和 microphone intent，但 draft 创建、append、finalize 和 discard 分别调用 `workspace:createSegmentAttachmentRecordingDraft`、`workspace:appendSegmentAttachmentRecordingAudioChunk`、`workspace:finalizeSegmentAttachmentRecordingDraft` 和 `workspace:discardSegmentAttachmentRecordingDraft`。Draft id 是 attachment id，不是同级 Segment id。Finalize request 必须携带 `workspaceId`、parent `memoryId`、parent `segmentId`、`attachmentId`、title 和 duration；成功 response 返回 memory summary、parent segment projection 和 attachment projection。Renderer 只用 memory summary 更新 Workspace snapshot，只用 parent segment projection 更新 Memory detail cache，对 attachment projection 不插入顶层 Segment strip。SegmentAttachment 录音 marker 使用 `targetKind: "segment-attachment"`、`parentSegmentId` 和 attachment id；保存恢复时调用 attachment finalize，放弃恢复时调用 attachment discard。当前 SegmentAttachment 恢复只提供保存或放弃，不提供继续检查回听，因为 draft audio preview IPC 只定义了 Segment draft read。
- Workspace Stage recording target flow：底部 `录音` 优先使用当前 Memory context 直接打开 existing-memory recording target；如果当前 snapshot 没有 Memory，则先打开 `MemoryCreateDialog`，创建成功后使用返回的 `memoryId` 打开 existing-memory recording target。创建失败停留在命名弹层，recording overlay 不打开，mic intent、draft directory、recording IPC 和 MediaRecorder 都不得启动。
- Memory create main flow：`workspace:createMemory` 校验 sender、handle ownership 和 request schema 后，由 main process 生成稳定 `memoryId`，在 `memories/<memoryId>/` 下 no-replace 创建空 Memory 容器，写入 `memory.json`，刷新 `.reo/index.json`，返回 Memory summary。Memory directory 已存在、`memories/` 不安全或 index 刷新失败时返回 `ERR_MEMORY_CREATE_FAILED`；如果已经写入 `memory.json`，普通失败必须删除本次刚创建的目录并重建 index，避免 index 指向缺失目录。Delayed lock usability 返回 lock lost 时必须停止后续写入、清理和 index 重写，直接返回 lock lost。
- Recording lifecycle 覆盖 idle、acquiring-permission、recording、paused、finalizing 和 failed；mic intent 拒绝、permission denial、media start failure 或 append failure 从忙碌状态进入 failed，discard 当前未完成 draft，不创建 finalized audio segment。Finalize failure 从 finalizing 进入 failed，并按 main error envelope 的 data retention 保留或清理 draft。idle 收到 late failed event 必须 no-op。
- Recording overlay 忙碌状态包括 acquiring-permission、recording、paused 和 finalizing；Drawer/Vaul close、Escape 和 overlay dismiss 必须阻止这些状态误关。App route 层把存在 recording target 视为录音流程正在占用当前工作区，阻止 sidebar 和 MemoryRail 引发的工作区切换、workspace release、移除、创建叠层或 current Memory 切换；overlay 同时把 busy close-blocked state 上报给 App，App 只在忙碌录音态通过 `beforeunload` 阻止原生窗口关闭或 reload，准备态不阻止原生关闭。
- 当前 recording overlay flow：loaded workspace frame 的 `录音` 优先使用当前 Memory context 作为 `existing-memory` target；没有当前 Memory 时先走 Workspace Stage recording target flow 创建 Memory，再以返回的 `memoryId` 使用 `existing-memory` target。Recording finalize request 必须携带 `memoryId`，main 只负责把 audio segment 追加到该 Memory，不从 recording 隐式创建 Memory。`开始录音` 先 await `beginMicrophoneIntent` 成功，再创建 recording draft，最后启动 browser MediaRecorder adapter；draft 创建失败、media start failure、stale session、组件 unmount 或记忆空间 handle 切换会 clear matching pending intent；draft 创建成功后如果 session 已 stale，会 discard draft 且不启动 media；只有 MediaRecorder controller ready 后 UI 才进入 recording 并显示 pause/stop。录音前、录音中和暂停态使用沉浸式 overlay，背景统一虚化弱化且不可作为当前操作区；录音前只允许开始录音，录音中允许暂停和完成，暂停态允许 cursor 定位、播放/暂停当前 draft、后退 15 秒、前进 15 秒、继续和替换。audio data 通过 append queue 串行写入；MediaRecorder adapter 以短 timeslice 输出 durable WebM/Opus audio chunks，通过 Web Audio analyser 输出低频 level samples 供 waveform 使用，并从同一麦克风 stream 通过 AudioWorklet 可选输出 16 kHz 16-bit mono little-endian PCM chunks；renderer 在 main `workspace:startRecordingTranscription` accepted 后通过有界串行队列把 PCM chunks 发送到 `workspace:sendRecordingTranscriptionAudio`，queue overflow、send failure、ASR start 未 accepted 或 ASR 失败只关闭 live ASR、标记 completion backfill 并用 root toast 告知，不得中断 durable capture。renderer 按有效录音时间保留当前 session 的 MediaRecorder chunk 前缀和 PCM chunks；暂停态回听只使用完整 MediaRecorder 前缀，ASR flush/backfill 会按 cursor 裁剪首个重叠 PCM chunk，不发送 cursor 前音频。pause 暂停 MediaRecorder、麦克风音轨、AudioWorklet PCM 输入、level pump 和 elapsed timer，drain 已排队 PCM 后 finish 当前 ASR 输入，并把 `cursorTimeMs` 设置到当前 `totalDurationMs`；暂停态播放从当前 `cursorTimeMs` 开始，audio `timeupdate` 更新 cursor、波形播放头和时间显示；回放 ended 把 cursor 放到末尾；拖动波形或前进/后退 15 秒会同步更新 audio currentTime。cursor 位于末尾时 resume 恢复当前 MediaRecorder controller 采集，并以当前 effective duration 作为 ASR timestamp offset 开启新的 main transcription session；cursor 位于中间且大于 0 时 replace 会先把 cursor 对齐到实际可保留的 durable chunk 边界，再创建新的 recording draft、按 cursor 前完整有效 chunk 前缀重建音频、截断 waveform、PCM 和 transcript timeline、生成新 `revisionId`，再复用当前已暂停的 MediaRecorder controller 从该位置继续采集，并用 replacement start time 作为 ASR offset 启动新的 main transcription session；cursor 位于起点时 replace 不克隆旧 draft prefix，改为创建新的 recording session、revision、draft 和 MediaRecorder controller 后从 0 重新采集；旧 draft 在新 draft 接管后丢弃，旧 session/revision 的异步结果必须被丢弃。stop 先停止 recorder，等待 MediaRecorder `stop` 事件和最后一次 `dataavailable` 转换完成，再等待 append queue 清空；随后 finish 当前 ASR 输入并在有界时间内等待最终转写包，最终转写成功或失败后才进入 finalize。最终转写失败只使用 root toast 告知，durable audio finalize 继续执行。Finalize audio 成功后必须先把完整 finalized audio Segment projection 写入 recovery marker，再保存 transcript；如果 transcript save 失败，后续恢复只补 transcript，不重复 finalize。Finalize 后关闭录音层，不弹出强制描述、转写编辑或反思编辑窗口；live transcript segments 或恢复 sidecar markdown 会形成 transcript draft，非空文本会通过 `workspace:saveTranscript` 保存，不生成本地 mock transcript；App 负责刷新 Workspace snapshot 的 `memories[]` summary、把 finalized audio Segment projection 合并到当前 Memory detail cache、把受影响 Memory 设为当前 Memory context，并用一次性 focus intent 选中新 Segment。Transcript save 成功后由 overlay 通知 App main 返回的 Memory summary、`memoryId` 和 `segmentId`；App 以 functional state/cache update 合并 summary，更新对应 Memory detail cache 的 transcript presence，并 invalidate 该 Segment content Query，不用 stale renderer 闭包覆盖并发保存结果。非忙碌状态关闭 overlay 会清空 renderer-local recording draft、error、elapsed、cursor、waveform samples、playback URL、recording target 和 machine state，忽略已关闭 session 的 transcript save/ASR 结果；下一次打开必须从 idle ready state 开始。
- 当前 recording boundary flow：录音有效时长小于 2 秒时，首次 stop 只给出中性短录音提示并保持当前录音数据，第二次 stop 才允许 finalize；持续低音量输入达到 15 秒时只显示一次中性提示，capture、append queue 和 ASR 输入继续；单次录音到 55 分钟显示接近上限提示，达到 60 分钟自动执行 pause flow 并把 cursor 设置到当前有效末尾；替换首次点击只提示覆盖影响，同一 cursor 再次点击才进入 replacement flow。replacement 不创建新的 microphone intent；replacement draft 创建、retained prefix copy 或 fresh capture restart 失败必须保留原 active draft，不能误导按钮状态或丢失原录音。
- cursor 大于 0 的替换 retained audio copy 由 main process 的 `workspace:cloneRecordingDraftPrefix` 完成：renderer 先等待 append queue 静止并创建 target draft，再请求 main 从 source draft 复制 cursor 前有效 byte prefix 到 target draft。Main 必须校验 source/target metadata、source retained byte length、target 空音频、同 draft write lock 和 workspace containment；复制循环中每个 read/write chunk、target fsync 和 metadata 写入前都必须重新消费 handle lock usability。copy 失败或 lock lost 必须把 target `audio.webm` 回滚为空并 fsync，renderer rollback 必须恢复旧 draft、captured chunks、PCM chunks、waveform、duration、cursor、revision、timeline 和 transcript draft。cursor 为 0 的替换不得调用 prefix clone，不得把无初始化 header 的旧 WebM 后段作为新音频；必须从新的 MediaRecorder controller 和新 recording session 开始。
- 当前 recording abnormal recovery flow：renderer 在 recording draft 创建后写入按 `workspaceId` 隔离的 local recovery marker，记录当前 `memoryId`、`segmentId`、title、有效 duration、append `nextSequence`、recording session、revision、waveform samples、audio chunk time/byte map、safe audio byte length 和当前有效 transcript segments；录音中 duration 至少每秒更新一次，首个 MediaRecorder chunk 必须写入恢复快照，后续 chunk map 按节流更新，ASR segment、pause/stop 前和 finalized audio 已保存但 transcript 尚未保存时会刷新恢复快照；append ack 不逐次写 localStorage，恢复继续检查时以 draft metadata 读取真实 `nextSequence`。marker 超出预算时可以裁剪 waveform 或压缩 audio chunk map，但 transcript 文本不得静默丢弃；必要时 transcript markdown 写入同 workspace 的 renderer recovery sidecar，marker 保存引用和 segment 校验。正常 finalize、主动 discard、replacement 丢弃旧 draft、组件 unmount 丢弃 draft 或关闭清理会删除匹配 marker 和 transcript sidecar。重新打开同一 workspace 后，App 只在 marker 的 `workspaceId` 匹配且 `memoryId` 仍存在于当前 snapshot 时显示“未完成录音”恢复对话框；用户选择保存时使用当前 `workspaceHandle`、marker 的 `memoryId`、`segmentId`、title 和 duration 调用 `workspace:finalizeRecordingDraft`，成功后合并 Memory summary 和完整 finalized audio Segment projection；如果 transcript 保存失败，App 将 finalized audio Segment projection 和最新 transcript snapshot 写回 marker 并保留恢复提示，后续保存重试只补 transcript，不把 `ERR_RECORDING_NOT_FOUND` 当作成功；audio finalize 和 transcript save 都完成后才清除 marker 并保持当前 Memory context。用户选择放弃时调用 `workspace:discardRecordingDraft`，成功后清除 marker；用户选择继续检查时 App 打开该 Memory 的 recording overlay，overlay 只有在 marker 存在 audio chunk byte map 时才通过 `workspace:readRecordingDraftAudio` 读取未完成 draft audio，读取上限来自 marker 的 chunk byte map，并按 marker 的 chunk map 还原暂停态 chunk 前缀、waveform、cursor、timeline 和 revision；如果 marker 只有 transcript sidecar markdown 而没有 transcript segments，overlay 必须把该 markdown 作为待保存 transcript draft。没有 marker-derived chunk map 时不得用固定大上限读取 draft audio，只允许保存或放弃。恢复后的暂停态可以直接完成保存、回听检查或放弃；由于没有原 MediaRecorder controller，继续录制和中段替换不会跨会话拼接 WebM，用户触发时只显示 root toast 并保留 marker 以便保存或放弃。恢复失败使用 root toast，marker 保留以便重试。
- 当前豆包流式语音识别完成 main 侧 live session helper 和 renderer subscription 边界：鉴权 header 由 main runtime 输入构造，资源 ID 是 `volc.seedasr.sauc.duration`，连接端点是 `wss://openspeech.bytedance.com/api/v3/sauc/bigmodel_async`，full request 使用 16kHz/16bit/mono raw PCM、`bigmodel`、`show_utterances`，audio-only request 使用 gzip 分包和正/负 sequence，server response 解析为带 `recordingSessionId` 和 `revisionId` 的 transcript segment；main `recordingTranscriptionSessions` 按 sender、workspaceHandle、recordingFlowSessionId 和 recordingSessionId 绑定 active session，替换或重新开始会关闭旧 entry，旧 revision audio 和旧异步 segment 返回都会被丢弃。Preload 只暴露显式 start/send/finish/close 方法和去除 `IpcRendererEvent` 的 `onRecordingTranscriptionEvent` callback；renderer 不持有 App ID、Access Token 或火山鉴权 header。Renderer live PCM 发送必须通过按 recording session / flow / revision 绑定的有界串行队列，队列 drain 期间如果 live session 已关闭则不再调用 finish，并改走 completion backfill；live PCM send 返回 `accepted:false` 时也必须关闭 live ASR 并标记 completion backfill，避免已有部分 transcript 时尾部音频静默掉队。ASR `start` 初始连接失败时由 main registry 重试一次，临时失败不会向 renderer 发送 transcript error event；重试仍失败时返回脱敏错误。录音中的 live session 发生 transport error 或非预期 close 时，main registry 会用同一 recording session 和 revision 启动新 session，保留并回放最近 5 秒 16 kHz PCM buffer；如果 replay 起点落在某个 PCM chunk 内，main 会裁剪首个重叠 chunk，避免重复发送 replay 起点之前的音频；回放段的 transcript timestamp 继续映射到原录音时间线；重连成功不向 renderer 发送临时 error event，重连耗尽后才发送脱敏 error event，durable capture 继续。Renderer 同步缓存当前有效 16 kHz PCM chunks；如果完成时实时 transcript 为空，或当前 revision 已收到 ASR error / finish failure，会先 finalize durable audio，再用同一 `recordingSessionId` / `revisionId` 和 `completion-backfill` recording flow session 将缓存 PCM 重新发送给 main ASR session，等待最终包后保存 transcript；补转写失败只使用 root toast 告知，durable audio 已保存结果不回滚。ASR `finish` 发送 final audio frame 后必须等待服务端 last package，收到最终 segment 后才确认 closed；最终包超时、socket 提前关闭或 finish 失败时返回安全错误，renderer 用 root toast 告知并继续保存 durable audio。close transcription 在 sender/handle owner 匹配后允许在 workspace lock lost 时清理 active session；finish transcription 在 lock lost 时必须先关闭 owner-matched active ASR session，再返回 lock lost，避免无 UI owner 的 live socket 残留。
- Audio append 必须按 sequence 串行 ack；每个 recording draft 只允许 1 个 append 在途，超量进入 failed recoverable。当前进程创建的 active draft 可以跳过每个 append 的 durable finalized scan；记忆空间 close、window teardown 或 root runtime state 清理后必须移除该进程态，后续 append 重新检查 finalized truth；stale/manual draft append 必须先检查 durable finalized audio segment。存在 finalized truth 时返回 finalized error，即使 `.reo/drafts/segments/<segmentId>/` 中仍有 stale draft。Append 写 chunk 前必须确认 draft `audio.webm` 是记忆空间内普通文件，拒绝 symlink 或非文件 audio；metadata 写失败时必须 truncate audio 回 append 前 size 并 fsync，返回 `ERR_RECORDING_APPEND_FAILED` 与 `dataRetention: "draft-preserved"`，且不得推进 sequence、audio bytes 或写到记忆空间外。
- MediaRecorder chunk 转换失败、append 返回错误、append promise reject 或 media start 失败时，recording overlay 立即进入 failed，不调用 finalize，停止当前 recorder controller，discard 当前未完成 draft，并用 recording session token 忽略 stale recorder 后续 chunks；discard draft cleanup 必须重新验证 workspace containment，不能跟随 symlink parent 删除 记忆空间外目录；failed 状态允许用户重新开始一次新的 recording draft，retry 会清空 stale elapsed、transcript draft 和 timeline state。
- Finalized audio segment 不允许继续 append。Finalize 必须先复核 `memories/` root，再逐段创建并复核 `memories/<memoryId>` 和 `memories/<memoryId>/segments/`，不能用递归 `mkdir` 跟随 symlink；创建 memory directory 和 segments directory 的目标解析后、实际 `mkdir` 前后都必须重新解析 workspace child containment，拒绝 ancestor symlink swap。随后在已验证 `segments/` parent 内使用相对 `mkdir` 创建 Reo-owned `.reo-finalizing-*` staging directory，不得用缓存 absolute path 创建 staging；staging 创建后写 marker、fsync、copy 和 expose 前都必须重新解析当前 staging directory；staging `segment.json` 读取必须使用 no-follow file handle。再通过完整 workspace child containment 重新解析 draft source，复核 `.reo/drafts/segments` ancestor 和 leaf draft directory 都不是 symlink 或非目录，复制前再次解析 draft source，只允许异步复制普通 `segment.json`、`audio.webm` 和 `transcript.md`，未知文件必须失败并保留 draft；复制后 fsync staging tree。在 expose 前必须重新通过完整 containment 解析 target segment path 并复核 staging parent，最终 expose 在已验证 parent directory 内重验 target missing，创建 target directory 作为 no-replace reservation，再先搬迁 marker、后搬迁 staging 内容，parent identity 改变或 late-created target 必须失败，不能使用已缓存 absolute path 直接暴露；最后 fsync parent。
- Finalize cleanup 在删除 draft、staging、target 或未提交 memory 目录前后必须重新做 workspace containment；如果 cleanup path 在验证后变为 unsafe，main 必须失败或跳过删除，不得删除 记忆空间外文件。
- Finalize 在 staging copy 后、durable target expose 前必须按 draft metadata schema 重新校验复制结果，并确认 `workspaceId`、`segmentId` 和 `audioByteLength` 与当前请求及实际 audio file 一致；该校验失败属于 pre-expose failure，必须清理 staging、保留 draft、不得删除 draft 或写出 finalized truth。
- Finalize 在 staging 暴露前失败时必须删除 `.reo-finalizing-*` staging tree；如果这是新建 memory 且尚未写入 `memory.json`，也必须删除该空 memory 目录，保证同一显式 `memoryId` 可以重试。
- Directory fsync 在不支持的平台或文件系统上按 best-effort 处理；`EACCES`、`EISDIR`、`EINVAL`、`ENOTSUP`、`EPERM` 不得让已成功写入的 atomic file 或 finalize transaction 误回滚。
- Finalize 使用 renderer active recording clock 计算的显式 duration 和实际 `audio.webm` 文件大小写入 `segment.json`、返回值和 `.reo/index.json`，三者的 `audioByteLength` 必须一致，且 `segment.json.status/memoryId/segmentId` 必须与目录 ownership 一致。`memory.json` 写入和 previous-memory rollback 前必须重新解析 memory directory，不能复用 final rename 前缓存的 memory path。
- Memory metadata、记忆空间 metadata、workspace index 和 finalized segment metadata 必须从记忆空间内普通文件读取；`.reo/workspace.json`、`.reo/index.json`、`memory.json` 和 finalized `segment.json` leaf 必须用 no-follow file handle 打开，symlink leaf 不进入 open、rebuild 或 lookup 投影。每个 metadata/index JSON 文件不得超过 1 MiB，schema 必须 strict，unknown 字段不进入当前合同。Finalized audio segment transcript save public request 必须携带 finalize response 返回的 `memoryId` 和 `segmentId`；main 直接解析 nested finalized segment directory，遇到 unsafe finalized metadata、ownership mismatch 或 duplicate durable segment id 必须返回 typed not-found/error envelope，不能 fallback 到 stale draft。markdown save 必须在当前 segment directory identity 内执行 atomic write，同一 root/memory/segment/file 进入 main process 写队列，避免 out-of-order save 旧内容覆盖新内容。
- Finalized audio file 必须是记忆空间内普通文件，不能是 symlink 或非文件；index rebuild、finalized lookup 和 finalized read 都必须拒绝 symlinked `audio.webm`。`workspace:readFinalizedAudioSegment` 只允许读取当前 handle 下的 explicit `memoryId` + `segmentId` finalized audio truth，返回 bounded audio bytes、audioByteLength 和 `transcript.md` 文本，不返回 raw path；读取前后必须校验 handle usability、duplicate finalized segment id、metadata ownership、audio byte length 和 segment directory identity。`workspace:readFinalizedAudioSegmentAttachment` 只允许读取当前 handle 下 explicit `memoryId` + `segmentId` + `attachmentId` 的 finalized attachment audio truth，返回 bounded audio bytes 和 audioByteLength，不返回 transcript 文本或 raw path；读取前后必须校验 parent segment directory identity、attachment metadata ownership 和 audio byte length。暂停态回听默认使用 renderer 当前录音会话持有的有效 chunk 前缀创建临时 Blob，异常恢复只通过 `workspace:readRecordingDraftAudio` 在 marker byte map 限制内读取未完成 draft audio 并按 marker chunk map 重建回听检查前缀，不跨 MediaRecorder session 继续或替换。`workspace:readRecordingDraftAudio` 与同 draft 的 append、finalize 和其他 read 互斥；finalize 开始前也必须拒绝 active draft audio read，读取后必须重新校验 metadata byte length 与读取上限，避免 append/read/finalize race 返回超出 marker 上限或删除正在恢复预览的 draft。
- Finalize 复制 draft 后只能补齐缺失的 `transcript.md`；如果 draft 中已有普通 `transcript.md`，必须保留内容，不能用空文件覆盖。
- Finalize 更新 `.reo/index.json` 失败时，main process 删除 interrupted staging/target directory、恢复 `memory.json` 并保留 draft，返回 `ERR_RECORDING_FINALIZE_FAILED` 和 `dataRetention: "draft-preserved"`；如果 target directory 已存在且不属于当前 finalize marker，不得删除。若 delayed handle usability 在 finalize 内部返回 lock lost，必须立即返回 lock lost，不得继续执行 stale handle cleanup、rollback 或 index 写入；后续 recovery 由新 owner 在重新获取 lock 后处理 marker/staging。
- Finalize 在 index 已刷新后又因 draft cleanup 前错误需要 rollback 时，必须删除 interrupted staging/target directory、恢复或删除 `memory.json`，并重新刷新 `.reo/index.json`，避免 projection 指向已回滚的 recording。
- Finalize 成功时先删除 draft 并 fsync draft parent，再重新解析 durable target directory、复核 containment、清除 durable target marker；如果 draft 删除前失败，返回 `dataRetention: "draft-preserved"`；如果 draft 已安全缺失，按 cleanup 已完成处理，不得回滚已写入的 durable audio segment；如果 draft 已删除但 draft parent fsync、target 复核或 marker 清除失败，返回 `dataRetention: "durable-marker-recovery-required"`，不能把该 finalized audio segment 当作已完整收口。删除 draft 后、draft parent fsync 和 marker unlink 前必须重新消费 delayed handle usability；中途 lock lost 时保留 durable marker，让后续新 owner recovery 收口。Recovery 看到 marker-bearing valid finalized audio segment 时也必须先删除 stale draft、确认 draft parent fsync、重新解析 target directory，再清除 marker；如果 draft 已安全缺失，按 cleanup 已完成处理，继续 fsync parent 并清 marker，避免 stale draft 影子读取或 durable marker 永久保留。open 调用 recovery 时，staging/partial 删除、draft cleanup、draft parent fsync、marker unlink、orphan segment cleanup 和 memory repair 写入前必须消费当前 lock/root/`.reo` usability；中途 lock lost 时停止 recovery 写操作并返回 lock lost。Recovery 遇到缺失 `memory.json` 但带 marker 且含 `segment.json` 或 `audio.webm` 的 finalized segment payload 时必须 fail-open 保留；删除 staging/partial segment 后，如果 orphan memory directory 没有 `memory.json` 且只剩 `segments/`，必须删除该 metadata-less memory directory，即使 `segments/` 内还有 markerless partial target，避免同一显式 memory id 重试被阻断。
- 同一 `memoryId` 的 create、append 和 title update 通过 main process memory write lock 串行保护；并发 append 中未获得锁的一方返回 `ERR_RECORDING_FINALIZE_FAILED` 和 `dataRetention: "draft-preserved"`，并发 title update 中未获得锁的一方返回错误信封并保留既有 `memory.json`。
- `.reo/index.json` 损坏、丢失或合法但陈旧时，open/rebuild 会扫描 `memories/*/memory.json`、finalized segment metadata 和 `audio.webm`，只从 schema、status、ownership 和 metadata/audio 一致的 finalized audio segments 协调 memory summary 并写回 index；scan 必须绑定 `memories/` root identity，scan 前、scan 后或持久化前 identity 改变时失败并保留原 index；如果 `memories/` 存在但不可读取，open 必须失败并保留原 index，不能把 read error 协调为空记忆空间。Transcript save 只刷新所属 memory 的 index entry，避免每次保存重扫全 workspace；同一 workspace 的全量 index replace 和 index entry refresh 进入同一个 main process write queue。全量 replace 与 open reconciliation replace 都在队列内重新读取当前 read model 后再写入；单条 refresh 在队列内部重新读取当前 memory summary，再按顺序读取当前 index、合并单条 memory summary、atomic 写回。
- Recovery 扫描 `memories/<memoryId>/segments/` 前必须重新做 workspace containment 和 symlink guard；如果 `segments` 本身或 `<segmentId>` leaf directory 是 symlink，不能跟随、不能删除 symlink target，只能按缺失 durable recording 修复当前 memory metadata 投影。Recovery repair 写回 `memory.json` 前必须重新解析当前 `memoryDirectory()`，不能复用扫描开始时缓存的 `memoriesDirectory`。
- 如果进程在 finalized metadata 写入后、`.reo/index.json` 写入前崩溃，后续 open 会从记忆空间文件真源恢复 `.reo/index.json` 的 memory summary，并让 Workspace snapshot 的 `memories[]` 投影重新可见。
- Metadata、index 和 transcript 的 atomic write 先异步写入并 fsync temp file，再在已验证 parent directory 内 rename；temp open、data write、rename 与 parent fsync 前后复核 parent directory identity。replace write 覆盖已有 target 前必须保留同目录 backup，backup 删除前的后置 validation 或 fsync 失败时恢复旧 target，不能删除唯一副本；backup 已删除后的 cleanup durability failure 属于 best-effort，不得把已提交 target 误报为失败；`AGENTS.md` no-replace write 在 temp file fsync 后 hard-link 到目标 path。当前 Node API 没有 fd-relative `renameat/linkat`，final commit 使用短同步 critical section 临时进入已验证 parent directory；该窗口内不得 `await`，也不得调用依赖 `process.cwd()` 的外部逻辑。若 commit 后发现 parent path identity 改变，必须删除本次暴露的 target；replace write 还必须恢复旧 target。parent directory fsync 成功时提供更强持久性，遇到当前平台或挂载点不支持 directory fsync 的错误时不得在 rename 或 hard link 后误报写入失败。
- `updateWorkspaceIndex` 在 update 函数成功前不会持久化 open/rebuild 协调结果；失败路径不改变已有合法 index。
- Finalize 必须等待最后一个 append ack，不得 duplicate stop 提前 finalize。
- Transcript save failure 使用 root toast 显示，durable audio finalize 结果保留，main atomic write 保留 previous disk file。
- Paused draft playback 不读取 finalized audio IPC，只用当前 renderer 已捕获的有效 MediaRecorder chunk 前缀创建临时 Blob URL；新 chunk 到达、替换、完成、关闭或 unmount 时必须停止回听并释放临时 URL。
- Memory space write 使用 single-writer lock；第二个 Reo writer 返回 locked error。带 owner 进程已退出、进程启动指纹不匹配、缺失 owner file、损坏 owner file 或 symlinked owner file 的 stale lock directory 可在下一次 acquire 时被替换；不含启动指纹的 owner file 通过 owner file mtime 与当前 pid 启动时间判断 PID 复用；无法确认 owner 是否仍是同一活进程时不做 silent override。

## 时序规则

- 多步骤 write 必须有 transaction boundary。
- 改变 cached data 的 mutation 必须有 invalidation 或 rollback。
- TanStack Query mutation 必须定义 query key、invalidation scope、pending/error state 和 rollback 策略。
- Zustand persist 必须定义 storage、version、partialize/migrate 和 user-visible recovery。
- Form submit 必须定义 validation timing、submit owner、failure behavior 和 reset behavior。
- Background task 必须有 owner、trigger、cancellation、retry、error reporting。
- Diagnostic/background error flow 必须有 owner、trigger、redaction、retention、retry 和 failure behavior。
- Session/auth change 必须写明 caller、persistence effect、renderer visibility、recovery behavior。
- Better Auth Electron flow 必须先定义 system-browser request、PKCE/state、callback exchange、session persistence、renderer visibility、sign-out、error 和 recovery。
- Packaging flow 必须先定义 build、package、sign、notarize、make、publish、verify 的顺序和 failure behavior。
- Update flow 必须先定义 trigger、channel、metadata source、download/install timing、cancellation、retry、rollback、error reporting 和 renderer visibility。
- Concurrency-sensitive flow 必须写 race condition analysis。

## 禁止

- 禁止 generic event runtime。
- 禁止没有当前产品需求的 mailbox 或 queue runtime。
- 禁止没有 observability 的 hidden background job。
- 禁止没有 signed packaged app 和 release metadata 的 updater polling。
- 禁止没有 rollback 的 optimistic UI path。
- 禁止依赖未记录 timing assumption 的 async flow。
