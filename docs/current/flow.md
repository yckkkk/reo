# 流程

本文档是行为时序、生命周期和恢复策略的当前真源。

## 当前事实

- 当前 IPC request flows 覆盖 `window.reoWorkspace` 的 choose、list memory space entries、initialize、open、open memory space entry、remove memory space entry、memory space title update、close、Workspace snapshot read、Memory create、Memory detail read、finalized audio segment read、finalized audio SegmentSupplement read、memory title update、Segment title update、SegmentSupplement title update、Memory delete/restore、Segment delete/restore、recording draft、SegmentSupplement recording draft、Segment transcript save、SegmentSupplement transcript save、microphone intent begin/clear、recording transcription start/send/finish/close、voice transcription settings read/setEnabled/save/clear/validate、受限外部链接打开，以及实体 More 菜单 shell 动作。
- 当前 preload/IPC consumer 覆盖记忆空间文件事务基础、create memory space UI flow、Memory create UI flow、MediaRecorder recording overlay 和 transcript save。
- 当前没有 auth/session lifecycle。
- 当前没有 auth request、exchange、sign-out 或 user-update flow。
- 当前记忆空间 file write 使用 atomic temp file、file fsync、rename 或 no-replace hard link、parent directory fsync 边界；可替换 metadata 用 rename，不能覆盖用户文件的 stable path 用 no-replace hard link。
- 当前 main process 的 workspace directory transaction primitives 位于 `src/main/workspaceDirectoryTransactions.ts`，覆盖当前目录内同步 critical section、best-effort current directory fsync、known-directory no-replace open、known-directory existing open、known-directory file remove、known-directory entry read、known-directory tree remove 和 known-directory empty directory remove。调用方仍负责具体 transaction 的 rollback、staging、recovery 和用户可见错误语义。
- 当前 directory rename transaction 不进入通用 helper。`memoryFiles.ts` 保留 finalize staging expose 的 same-parent directory move，因为它先创建 target、先移动 finalize marker、逐项移动 payload，并在部分移动失败时只清理已验证 target identity；`memoryFiles.ts` 的 file-space node move 保留 source/target parent identity、可选 expected source identity、target preflight、lock recheck 和双 parent fsync；`workspaceFiles.ts` 的 workspace root rename 保留 platform no-replace move、nested-move repair、canonical root finalization 和 stale projection error split；`workspaceLock.ts` 的 stale lock cleanup 保留 owner liveness check、lock directory identity compare 和 release-time ownership validation。
- 当前没有 DB migration 或 startup database lifecycle。
- 当前没有 background queue。
- 当前 Query flow 覆盖 app 启动读取 memory space list、记忆空间 initialization/open 后 invalidate memory space list、memory space ready 后 seed 记忆空间 snapshot cache、选中 Memory 后读取 Memory detail cache、选中 Segment 后读取 Segment content cache、切换到 selected Segment 的真实 SegmentSupplement tab 后读取该 supplement 的 content cache，以及读取 application-scoped voice transcription settings。Snapshot cache 只承载记忆空间身份和 `memories[]` summary 投影；Memory detail cache 只承载当前 Memory 的 finalized audio Segment projection 和每个 Segment 的 finalized supplements projection；Segment content cache 只承载 selected finalized audio Segment 的 audio bytes 和 transcript 文本；SegmentSupplement content cache 只承载 selected Segment 下单个 finalized audio supplement 的 audio bytes、audioByteLength 和 transcript（exists 与 text）；Voice settings cache 只承载 `enabled`、`apiKeyConfigured`、`apiKeyLastFour`、`lastValidatedAt`、`lastValidationOk` 和 `lastValidationCode`，不绑定 workspace session；Memory space、Memory、Segment 和 SegmentSupplement rename 使用 scoped optimistic update，Memory delete/restore 使用 response 后 cache 合并和 query 移除，Segment delete 使用 renderer scoped optimistic projection、toast undo rollback 和 delayed IPC commit，其它 mutation 不写 optimistic state。
- 当前 App route flow 使用 renderer in-memory state。无 active memory space 时显示 starter Home shell；memory space ready 后显示 loaded workspace frame；`首页` 或 `资料库` 在 active memory space 存在时先调用 `closeWorkspace` 释放当前 handle，再进入对应顶层页面。
- Loaded workspace frame 的状态 owner 分层：App 持有顶层 view、active workspace session、当前 Memory id、recording flow 和跨 route 阻塞；WorkspaceFrame 持有主内容轨道、MemoryRail 展开意图和 inline/overlay rail layout；Memory Studio 持有当前 Memory 内的 selected Segment、预览流、时间轴和内容 tab；RecordingOverlay 持有 recording machine、MediaRecorder controller、draft chunks、paused preview、replacement transaction 和 finalize owner。
- 录音流程在 App 中是单一 `recordingFlow`：closed 或 active。active flow 包含 target、visible open 状态、close-blocked 状态和可选 recovery review draft。流程打开后，App 阻止顶层导航、创建或打开其他记忆空间、移除或重命名记忆空间、切换当前 Memory context 和进入 settings mode；被阻止的 action 不得调用 open/close/remove/update workspace IPC，也不得清空当前 recording flow。点击完成会先隐藏 visible surface，但 active flow 保留到 finalize、transcript save 和 recovery marker 收口完成；失败时同一 flow 重新显示失败恢复态。
- Finalize 后 App seed 记忆空间 snapshot cache 的 `memories[]` summary，把受影响 Memory 设为当前 Memory context，并把完整 finalized audio Segment projection upsert 到对应 Memory detail cache；SegmentSupplement finalize 后 App seed Workspace snapshot summary，并更新当前 Memory detail cache 的 parent Segment projection；Memory detail 内的 Segment projection 按投影 `updatedAt` 倒序排列；recording transcript save 成功 response 返回当前 Memory summary，App 以该 summary 更新 Workspace snapshot cache 和当前 session state，并 invalidate 对应 finalized content Query；SegmentSupplement transcript save 成功 response 返回当前 Memory summary、parent Segment projection 和 supplement projection，App 更新 Workspace snapshot cache、当前 session state 和当前 Memory detail cache，并 invalidate 该 supplement 的 exact SegmentSupplement content Query；该 route state 不写入 URL、记忆空间文件、Query key 或持久化 store。
- 当前 memory create flow 是 request/response mutation：Workspace titlebar `新建记忆` 或空 Memory 状态下的 `录音` 打开 `MemoryCreateDialog`，React Hook Form 持有提交前 title draft；renderer 调用 `workspace:createMemory`，main 通过 workspace handle 校验 sender 和 lock usability 后进入同 `memoryId` write lock，创建 Memory 容器，写入 `memory.md` 与 `.reo/objects/memories/<memoryId>.json` 并刷新可重建 index，返回 Memory summary。Renderer 只在成功 response 后合并当前 session snapshot、seed snapshot cache，并把新 Memory 设为当前 Memory context；若 intent 是录音，App 先关闭 create intent，再以新 Memory 启动 recording flow；失败时 Dialog 保持打开并使用 root toast 显示可恢复错误，不写 optimistic state。
- 当前 memory rename flow 是 optimistic request/response mutation：MemoryRail More 菜单打开 `MemoryRenameDialog`，React Hook Form 持有提交前 title draft；提交后 renderer 立即关闭 Dialog、更新当前 session snapshot 和 snapshot Query cache。main 通过 workspace handle 校验 sender 和 lock usability 后进入同 `memoryId` write lock，按 `.reo/objects/memories/<memoryId>.json` 找到 Memory 目录，把目录 basename 更新为 `<memoryId>--<title>`，写入 `memory.md` title frontmatter，刷新可重建 index，并返回 Memory summary。title rename 不改变 activity `updatedAt`。保存失败时 renderer 只在当前 title 仍是本次提交值时回滚，并使用 root toast 显示可恢复错误。
- 当前 memory delete/restore flow 是 request/response mutation：MemoryRail More 菜单打开 `MemoryDeleteDialog`，确认前不触发文件写入；确认后 renderer 调用 `workspace:deleteMemory`，main 通过 workspace handle 校验 sender、request schema 和 lock usability 后，按 metadata id 找到 active Memory 目录并移入 `.reo/trash/memories/`，刷新 `.reo/index.json`，返回刷新后的 `memories[]` 和 `restoreToken`。Renderer 只在成功 response 后合并当前 session snapshot 和 snapshot Query cache、移除被删除 Memory 的 detail cache，并在 toast action 上保留本次 `restoreToken`；如果当前 Memory 被删，renderer 选择剩余第一条 Memory 或回到 Workspace Stage。用户点击 toast 恢复时 renderer 调用 `workspace:restoreDeletedMemory`，main 用 `restoreToken` 作为 metadata id 找到 trash Memory 目录并移回 active `memories/` 后刷新 index，renderer 合并 snapshot cache、选中恢复 Memory 并显示恢复成功 toast。删除或恢复失败只显示 root toast，不写 optimistic state，不清空原 projection。
- 当前 Segment delete flow 是 renderer optimistic projection + delayed request/response mutation：Segment card More 菜单打开 `SegmentDeleteDialog`，确认前不触发文件写入；确认后 renderer 立即关闭 Dialog，从 Workspace snapshot cache、Memory detail cache 和当前 session projection 中移除该 Segment。App 显示可撤销 Sonner toast，10 秒 duration 是唯一 grace-period clock；toast 的 `恢复` action 使用统一 icon+文字 action 结构，toast 无 close button 且不可手动 dismiss，点击恢复只把目标 Segment 重新插回当前投影，并通过一次性 Segment focus intent 选中恢复 Segment，不调用 `workspace:restoreDeletedSegment`。每个 Segment delete toast 只有 `pending → committing|undone → settled` 单向 phase；重复 auto-close 不得重复提交，commit 已开始后点击恢复不再回滚 renderer projection。Grace period 内若 Workspace file-truth refresh 返回尚未删除的文件真源，renderer 在写入 snapshot cache 和 session 前重放 pending Segment exclusion；带 pending Segment 的 refresh 必须读取 identity-bearing Memory detail 后再投影，不能只按 summary aggregate 重复扣减；refresh 产生的 detail cache update 必须在 request id 和 session revision 仍匹配当前 session 后才写入。refresh 只保护目标 Memory detail、目标 Segment content 和目标 SegmentSupplement content，其它同 workspace 查询照常 invalidate。toast 自动关闭时，只有当前 workspace session 仍匹配原 handle，renderer 才调用 `workspace:deleteSegment`；main 通过 workspace handle 校验 sender、request schema、`workspaceId` 和 lock usability 后，按 `.reo/objects/segments/<segmentId>.json` 找到 parent Memory 下的 active Segment 目录，并把已验证 directory identity 传入最终 rename，把目录移入 `.reo/trash/segments/`；若验证后源目录被替换，main 返回 unsafe typed error，保留 active/trash 文件，不移动替换目录。成功 move 随目录保留 supplements 子树，刷新 `.reo/index.json` 中的父 Memory summary，并返回父 Memory summary、`segmentId` 和 `restoreToken`。Response 返回后 renderer 会再次校验当前 session handle；旧 handle 的 in-flight delayed response 不得改写新 session 或同 `workspaceId` 的新 cache。成功 response 先清除本次 pending，再对 main 返回的父 Memory summary 重放仍未提交的 pending Segment projections，然后合并当前 session snapshot 和 snapshot Query cache，并移除该 Segment content Query cache 及 `['workspace', 'segment-supplement-content', workspaceId, memoryId, segmentId]` 前缀下的 supplement content cache；delayed delete 的 `previous-file-preserved` 失败使用同一局部恢复路径并显示 root toast；`dataRetention: "file-written-index-stale"` 代表文件真源已写出但 index 或 handle 收口未完成，renderer 保持删除投影、不做本地恢复，并显示 root toast。`workspace:restoreDeletedSegment` 仍是 main 侧恢复区能力：它用 `restoreToken` 作为 Segment manifest id 找到 trash Segment 目录，只有 parent Memory 仍存在时才移回 active `segments/`，restore move 同样以已验证 trash directory identity 作为 expected source identity；Parent Memory 缺失时返回 typed error，trash Segment 保持可恢复状态。
- 当前 optimistic update 用于命名类 mutation 和 Segment delete renderer projection；命名 rollback 必须检查当前 UI 仍停留在本次 optimistic title，避免覆盖后续用户输入；Segment delete rollback 只恢复目标 Segment，不写回整份旧 snapshot，不重建已保留的 content cache；多个 pending Segment delete 必须按当前仍 pending 的集合重放 projection，不能让单个 delayed response 复活其他待删除 Segment。Pending replay 只按 pending Segment identity 和 additive delta 调整 summary；Workspace snapshot summary 没有 Segment ids，不用 `segmentCount` 这类 aggregate equality 判断实体是否已被删除。
- 实体 More 菜单 shell 动作（`reveal` / `open` / `copy`）是只读系统调用，不进入 lock-bound mutation 序列，不具备 transaction、rollback 或 grace-period 行为，也不写入 Query cache、session projection 或 registry projection。已打开记忆空间的 12 个 channel 在解析路径和调用系统 API 前仍必须完成 `requireHandle`、`workspaceId` 匹配和 lock usability（`assertWorkspaceUsable`）校验，避免失效 handle 或 lock lost 状态把路径交给系统 API；只接收 `workspaceId` 的记忆空间条目 channel 不要求记忆空间已打开，但仍按当前实现校验 trusted sender、registry entry 与 root ownership 后才解析路径。
- 当前 memory space entry flow 覆盖 create 记忆空间 title/description/folder selection validation、创建时在所选父目录下新建同名记忆空间文件夹、sidebar 持续显示已导入记忆空间、sidebar `打开本地记忆空间` 打开现有 Reo 记忆空间或初始化空文件夹、点击已导入记忆空间打开该记忆空间、sidebar 记忆空间更多菜单移除 registry entry、open failure 可见错误反馈、submit failure 保留 create draft 和成功后 route state 切换。
- 当前没有 persisted client-state migration flow。
- 当前 main process 有最小 fatal exception path：`uncaughtException` 写入本地诊断日志和 `console.error`，先释放当前记忆空间 handles，再退出。
- 当前 structured diagnostic lifecycle 只在 main process 内运行：app startup、renderer process gone、fatal exception 和 workspace IPC request completion 写入本地 `electron-log` 文件。Workspace IPC 诊断以 request start/finish span 表达，finish 记录 status 和 duration；诊断字段经过脱敏，不记录 root path、file path、display path、title、token、handle、payload、transcript、正文或 secret。诊断字段默认不展开对象、数组或未知字符串；channel、status、errorName、errorCode、mode、processType、reason 和 dataRetention 只在闭合 allowlist 中保留。当前没有 background error reporting flow、renderer/preload logging bridge、诊断 IPC 或远程 telemetry。
- 当前没有 package、make、publish、release 或 update lifecycle。

## 技术方向

- 多步骤行为先建模，再实现。
- Command/event 名称必须具体、贴近 domain。
- 在没有真实 queue/stream 需求前，优先 request/response。
- 新 preload/IPC flow 只能跟随真实 renderer-to-main 特权需求创建。
- 在没有真实 schema 和 migration owner 前，不创建 DB migration flow。
- 每个 lifecycle transition 都必须有 owner。
- failure、cancellation、retry、recovery 行为必须明确。

## 状态管理纪律

- 状态机必须先描述 domain lifecycle，再决定使用 component state、reducer、React Hook Form、TanStack Query 或 Zustand。
- 不得把同一状态同时放入 component state、cache、form draft 和 metadata。
- UI 状态、form 状态、main-backed async data、durable file state 和 derived state 必须分开定义 owner。
- 复杂状态优先用 feature-local reducer 表达转移；只有跨 component subtree 且有真实 owner 时才引入 Zustand。
- 每个 state transition 必须说明触发事件、允许来源、失败行为、用户可见状态和验证路径。
- State ownership 必须用矩阵表达；每个 state row 只能有一个 source-of-truth owner，重复投影必须写同步方向和冲突处理。

## 当前流程决策

- Memory space directory selection flow：renderer 调用 `window.reoWorkspace.chooseDirectory()`；preload 调用 `workspace:chooseDirectory`；main 校验 channel、main frame、trusted URL 和 session；OS dialog canceled 返回 canceled；selected 返回 `selectionToken` 和 `displayPath`；真实路径只保存在 main 的 selection token store。
- Selection token lifecycle：issued、consumed、expired、sender-mismatch、not-found；成功 consume 和 expired 会删除 token，错误 sender 不删除 token；所有错误都不返回 path。
- Memory space initialize flow：renderer 传 selection token、title、description；main 校验 sender、token 和安全 title，把 token 指向的目录作为父目录，在父目录下 no-replace 创建同名 child memory space root；child 已存在返回 `ERR_WORKSPACE_ALREADY_EXISTS` 且不写入记忆空间文件。child 创建成功后获取 single-writer lock 并初始化 Reo 文件。lock file 写入前和打开前必须复核 `.reo` directory identity，lock target leaf 必须用 no-follow 打开并拒绝 symlink，lock directory 必须在同一 `.reo` 当前目录内创建并绑定 directory identity；owner file 必须在 lock directory 内 no-follow 创建、写入 main process pid 与进程启动指纹并 fsync。owner 进程已不存在、进程启动指纹不匹配，或 owner file 缺失、损坏、symlinked 的 stale lock 可被替换；无法读取进程启动时间且 owner pid 仍存活时返回 locked error。`AGENTS.md` 通过 no-replace atomic write 创建；写入记忆空间文件 后返回 opaque `workspaceHandle`、`workspaceId` 和 snapshot。
- Memory space open flow：renderer 传 selection token；main 校验 sender 和 token，canonicalize selected root；如果该目录是现有 Reo 记忆空间，则校验 metadata 和 unsafe children 后获取 single-writer lock 并打开；如果该目录为空，则获取 lock 后再次确认 root 除 lock-only `.reo` artifacts 外仍为空，再原地初始化为 Reo 记忆空间，title 使用 selected root basename，description 为空；如果第二次确认发现目录已变成非空非 Reo，返回 metadata invalid 并清理 lock-only `.reo`。非空非 Reo 目录、unsafe `.reo`、unsafe `.reo/drafts/segments`、unsafe `memories/` 或 invalid 记忆空间 metadata 返回错误且不得留下 lock。lock file 写入前和打开前必须复核 `.reo` directory identity，lock target leaf 必须用 no-follow 打开并拒绝 symlink，lock directory 必须在同一 `.reo` 当前目录内创建并绑定 directory identity；owner file 必须在 lock directory 内 no-follow 创建、写入 main process pid 与进程启动指纹并 fsync。owner 进程已不存在、进程启动指纹不匹配，或 owner file 缺失、损坏、symlinked 的 stale lock 可被替换；无法读取进程启动时间且 owner pid 仍存活时返回 locked error。corrupt index 可重建，corrupt metadata 阻断写入且不得留下 lock。
- Memory space registry flow：initialize/open 在记忆空间文件成功返回 snapshot 后、handle 注册前，把 canonical root 与 snapshot 写入 main-owned registry；registry upsert 失败时释放记忆空间 lock 并返回 `ERR_WORKSPACE_MEMORY_SPACE_REGISTRY_WRITE_FAILED`。Registry upsert 按单 registry 实例串行执行，避免并发导入互相覆盖。Registry 写入使用 atomic JSON write；读取使用 no-follow file handle、大小上限和 schema 校验。
- Memory space list flow：renderer 通过 TanStack Query 调用 `workspace:listMemorySpaces`；main 校验 sender、channel 和空 payload 后只读取 main-owned registry file，返回不含 root path 的记忆空间元数据。Registry 文件缺失、损坏、schema 不匹配或 symlink leaf 返回空记忆空间列表；不可读 IO 错误返回 `ERR_WORKSPACE_MEMORY_SPACE_REGISTRY_READ_FAILED`，renderer 使用 root toast 显示错误。Registry 写入最多保留最近 100 个记忆空间，title/description display text 有上限；registry 只是记忆空间列表投影，不是记忆空间文件真源。List 不扫描每个记忆空间 root，不读取 `.reo/workspace.json`，不做同父目录 rename scan。
- Memory space registry open flow：renderer 点击 sidebar 记忆空间时只发送 `workspaceId` 到 `workspace:openMemorySpace`。main 只 resolve 该 `workspaceId` 对应的 registry entry；stored root 可读且 metadata `workspaceId` 匹配时使用该 root，stored root 不存在时才在原父目录下最多检查 200 个直接子目录寻找同一 `workspaceId`。未找到返回 `ERR_WORKSPACE_MEMORY_SPACE_NOT_FOUND`；root folder 已被用户删除且没有同父目录同 `workspaceId` 记忆空间时返回 `ERR_WORKSPACE_ROOT_MISSING`；找到后获取 single-writer lock，在任何 recovery 或 `.reo/index.json` rebuild/persist 前确认打开的 `workspaceId` 匹配请求，并把 root folder basename 写回 `.reo/workspace.json.title` mirror；随后复用记忆空间文件 recovery、valid index read、registry upsert 和 handle registration，返回新的 opaque `workspaceHandle`。Stale registry root 已变成其他 Reo workspace 时返回 metadata invalid，不写入该 workspace。该 flow 不使用 selection token，也不向 renderer 暴露 root path。
- Memory space registry remove flow：renderer 在 sidebar 记忆空间更多菜单中确认“移除记忆空间”后，只发送 `workspaceId` 到 `workspace:removeMemorySpace`。main 只从 registry 删除 entry，不解析、不打开、不删除本地记忆空间文件夹；缺失 entry 也按移除完成处理。移除 active memory space entry 时，registry removal 是主路径；renderer 在移除成功后回到 starter Home，并尽力调用 `closeWorkspace` 释放当前 handle。close 失败不得恢复 registry entry 或阻断列表移除，只通过 toast 显示当前会话释放失败。
- Memory space title update flow：renderer 从 titlebar Breadcrumb 的记忆空间 DropdownMenu 或 sidebar 记忆空间 More 菜单打开重命名 Dialog。Active 记忆空间提交 `workspaceHandle` 和 trimmed safe folder title 到 `workspace:updateMemorySpaceTitle`；inactive registry entry 提交 `workspaceId` 和 trimmed safe folder title。提交后 renderer 立即关闭 Dialog、更新当前 session snapshot 和 memory space list 投影。main 校验 sender 和 request schema；active path 复用当前 handle、lock usability 和记忆空间 file truth，在 single-writer lock 下先把 root folder basename 改成 title，再把当前 handle 与 lock 迁移到新的 canonical root，随后写入 `.reo/workspace.json.title` mirror；root move 与 metadata mirror 提交后才读取或重建 `.reo/index.json` 以返回 Workspace snapshot，root move 前的同名冲突不得改变 index；registry projection 只做 best-effort。inactive path 从 registry 解析 root、获取临时 single-writer lock、移动 root folder、迁移临时 lock root、写入 `.reo/workspace.json.title` mirror，再 upsert registry title/description/root projection；addedAt、lastOpenedAt 和列表顺序不变。目标 sibling folder 已存在时返回 `ERR_WORKSPACE_ALREADY_EXISTS`，不改 root folder、metadata、registry 或 index。Root move 成功后的 metadata/projection 失败返回 `file-written-index-stale`；renderer 对该错误保持 optimistic title，只显示 root toast，其它保存失败才在当前 title 仍是本次提交值时回滚。
- Workspace snapshot refresh flow：active 记忆空间 session 存在且没有录音流程打开时，renderer 在 session ready 后立即调用一次 `workspace:readWorkspaceSnapshot(workspaceHandle)`，用合法 index 启动后的 UI 尽快协调外部文件修改；该次静默刷新失败不显示错误，避免进入记忆空间时闪烁。之后 renderer 只在 document visibility 变为 visible 时调用同一 refresh；普通 window focus 不触发 refresh。main 校验 sender、handle ownership 和 lock usability 后，先按 root folder basename 修复 `.reo/workspace.json.title` mirror，再从当前文件 read model 重新读取 Workspace snapshot，并在需要时按文件真源协调 `.reo/index.json`。如果 response 与当前 snapshot 完全一致，renderer 只清除 refresh error，不 seed snapshot、不更新 session、不 invalidate memory space list、Memory detail、Segment content 或 SegmentSupplement content Query。发生变化时 renderer seed `['workspace', 'snapshot', workspaceId]`、更新当前 session snapshot、保留仍存在的 current Memory selection，否则选择第一条 Memory 或回到 Workspace Stage，并 invalidate 相关 Query。该 flow 不创建 file watcher，不把外部合法 Markdown/frontmatter 或 Reo manifest 修改当作错误；非法 frontmatter、identity 不匹配或 unsafe path 使用 typed error 反馈并保留现有 UI 文件状态。录音流程打开时不执行 refresh，避免外部文件变化绕过当前录音的 Memory/Workspace 切换保护。由于 Query key 不包含 `workspaceHandle`，renderer 在 ready 新 workspace session 前和 close 当前 workspace 成功后，必须清理同 `workspaceId` 下的 Memory detail、Segment content 和 SegmentSupplement content queries，避免同一 `workspaceId` 重新打开后复用旧 handle 的内容缓存。
- Memory space initialize/open 在获取 lock 后，如果记忆空间文件 write、recovery、index read/rebuild、handle registration 或 response parse 抛出异常，必须释放 lock 并返回 workspace error envelope；lock target 写入失败必须返回 `ERR_WORKSPACE_LOCK_FAILED`，不得让 IPC handler reject。Lock 可用性同时要求 root identity、`.reo` identity 和当前 lock directory identity 未变化；initialize/open 获取 lock 后、写入或打开记忆空间文件前必须再次检查 lock usability；initialize 文件阶段在补齐托管目录和写入 `AGENTS.md`、记忆空间 metadata、index 前都必须重新消费 lock usability；open 文件阶段在 target revalidation、metadata read、补齐托管目录、recovery、valid index read、必要 index rebuild 和返回 session 前也必须重新消费 lock usability；Reo 发起的 memory space root rename 保持同一 root 和 `.reo` directory identity，只把 lock 与 handle 的 canonical root 迁移到新 path；identity 变化或 release 失败后，handle require 和 delayed handler operation 必须返回 lock lost，不得继续授权写入。
- Memory space close flow：renderer 传 `workspaceHandle`；main 校验 sender ownership，释放 lock 并撤销 handle。Handle close 只有在 lock release 成功后才能删除 handle；release 失败时保留 handle 并返回 `ERR_WORKSPACE_LOCK_FAILED`。BrowserWindow `closed`、`render-process-gone`、trusted main-frame non-same-document navigation 和 trusted main-frame reload 都会尝试释放当前 main process 持有的记忆空间 handles；same-document navigation、subframe navigation 和 untrusted navigation 不释放 handle。Navigation 和 window teardown 的 close-all 通过同一 lifecycle helper 合并重入释放；teardown close-all 只删除 release 成功的 handle，release 失败的 handle 保留为 lock lost 状态以便后续 close-all 重试。
- Memory space lifecycle 覆盖 none、creating、ready、missing、conflict、unsupported、failed。
- Memory space entry lifecycle 覆盖 idle、menu-open、create-dialog-open、folder-selecting、canceled、creating、open-local-memory-space-selecting、create-error、open-local-memory-space-error 和 ready。
- Memory space session replacement flow：同一个 renderer 窗口只显示一个 active 记忆空间 session。Create/open-local/open-memory-space 成功拿到新 `workspaceHandle` 后，如果已有 active memory space，renderer 必须先调用 `closeWorkspace` 释放既有 handle；既有 handle 释放失败时保留当前记忆空间 UI、使用 root toast 显示错误，并尽力关闭新 handle，避免双 lock 泄漏。打开 promise reject 时必须使用 root toast 显示可见错误。
- Memory space creation renderer flow：`创建本地记忆空间` 打开 `WorkspaceCreateDialog`；title、description、selection token 和 displayPath 属于当前 React Hook Form lifecycle；所选目录是 parent location，main 在 parent 下创建 title 同名记忆空间文件夹；OS dialog canceled 不修改 draft 并把 focus 返回 folder picker；同名 child 记忆空间文件夹已存在时使用 root toast 显示创建失败，保留 title/description，清除已消费的 selection token 和 displayPath；initialize success seed 记忆空间 snapshot Query cache、关闭创建弹层并设置当前 renderer session state。
- Memory space open-local renderer flow：sidebar `打开本地记忆空间` 单独调用 folder picker；OS dialog canceled 不改变 route state；selected folder 可以是现有 Reo 记忆空间或空文件夹，空文件夹会原地初始化为 Reo 记忆空间；renderer 只发送 selection token 给 `openWorkspace`，不发送 title、description、displayPath 或 raw path；open failure 使用 root toast 显示，不打开创建弹层、不清空当前记忆空间 session；open success 先按 记忆空间 session replacement flow 释放既有 handle，再 seed 记忆空间 snapshot Query cache、invalidate memory space list 并设置当前 renderer session state。该动作不进入 `WorkspaceCreateDialog`，也不路由到 `initializeWorkspace`。
- Memory create renderer flow：Workspace titlebar 的 `新建记忆` 打开 `MemoryCreateDialog`；title draft 属于当前 React Hook Form lifecycle；submit 只发送 `workspaceHandle` 和 trimmed title 到 `workspace:createMemory`。成功 response 合并当前 session snapshot、seed snapshot Query cache、把新 Memory 设为当前 Memory context、关闭弹层并显示该 Memory 的 Studio 基础态。创建失败使用 root toast 显示可见错误，不写 optimistic Memory card。
- Memory delete renderer flow：MemoryRail More 菜单的 `删除` 打开 `MemoryDeleteDialog`；Dialog open state 属于 App component state。确认按钮 pending 时禁止重复提交；取消或关闭不改变 Workspace snapshot。删除成功 response 是唯一 UI 更新来源，renderer 用 immutable update 替换当前 `memories[]` 投影、移除被删 Memory 的 detail cache，并根据当前 selection 决定切换到剩余第一条 Memory 或 Workspace Stage。toast action 的 `恢复` 调用 restore flow；toast 之外当前不提供 trash browser。
- Memory delete/restore main flow：`workspace:deleteMemory` 和 `workspace:restoreDeletedMemory` 都必须在当前 handle owner、workspace root identity、`.reo` identity 和 lock usability 有效时执行。Delete move、restore move、index refresh 和 rollback 都只能在已验证 workspace child path 内进行；`.reo/trash/memories/` 缺失时按需创建，任何 symlink、非目录、ancestor swap 或 active/trash directory conflict 都返回 typed error envelope。Delete 和 restore 都按 `.reo/objects/memories/<memoryId>.json` 定位目录，目录 basename 可以来自 Reo 命名或 Finder 直接重命名。Delete 在 move 成功但 index refresh 失败时尝试移回 active memory directory；restore 在 move 成功但 index refresh 失败时尝试移回 trash directory。Rollback 失败时返回错误并保留文件真源，不向 renderer 报告已完成删除或恢复。
- Memory detail renderer flow：选中 Memory 后，Memory Studio 使用 `['workspace', 'memory-detail', workspaceId, memoryId]` Query key 调用 `workspace:readMemoryDetail`。Request 携带 `workspaceHandle`、`workspaceId`、`memoryId` 和 requestId；main 校验 handle owner、workspaceId 与 handle workspace 匹配、lock usability 和 Memory 文件真源后，从 matching Memory 目录下合法 finalized Segment 文件空间节点投影 Memory detail；Segment 是否存在不依赖可重建 mirror，返回同 requestId 的 Memory detail projection。Renderer 只接受 requestId、workspaceId 和 memoryId 与当前 Query 匹配的 response；不匹配视为 stale response，不写当前 UI。
- Segment selection renderer flow：Memory detail loaded 且存在 Segment 时，Memory Studio 默认选择当前 detail projection 的首个 Segment。普通录音 finalize 后，App 把完整 Segment projection 合并到当前 Memory detail cache，并传入一次性 Segment focus intent；Memory Studio 只在该 Segment 已进入当前 detail projection 后消费 intent 并选中该 Segment。用户点击 Segment 横向预览 item 的选择按钮时，只更新 feature-local `selectedSegmentId`，并同步当前 card、时间轴点和内容区；该 item 内含录音卡片、timeline 圆点、时间标签和 hover/focus 才显示的 More 菜单。选择不写入文件、URL、Query key、Workspace snapshot 或 registry。切换 Memory 会重置 Memory Studio component state，避免既有 Segment selection 留到新的 Memory。
- Segment strip carousel renderer flow：Memory Studio 的 Segment 横向预览流通过 feature-local DOM ref 读取 `scrollWidth`、`clientWidth` 和 `scrollLeft`，在初始化、scroll、ResizeObserver 和 window resize 后重新计算左右 `CarouselArrowButton` 显示状态。Segment item 是该 scroll owner 的直接子项，item 宽度使用紧凑 `--memory-studio-segment-card-size`，card、圆点和时间标签共同随同一横向滚动位移。不可达方向不渲染按钮，因此不可点击也不可被键盘聚焦。点击左右按钮只调用该横向预览流的即时滚动，不更新 `selectedSegmentId`、Query cache、Workspace snapshot 或 durable file。横向溢出只属于 Segment strip scroll owner，不改变 document scroll，也不撑开 WorkspaceFrame。
- Finalized audio Segment content renderer flow：Memory detail loaded 且 selected Segment 存在时，Memory Studio 使用 `['workspace', 'segment-content', workspaceId, memoryId, segmentId]` Query key 调用 `workspace:readFinalizedAudioSegment`。Renderer 只接受 requestId、workspaceId、memoryId 和 segmentId 与当前 Query 匹配的 response；不匹配视为 stale response。Query 成功后，Memory Studio 用 audio bytes 创建 local Blob URL，`播放片段` 调用 hidden HTMLAudioElement `play()`，`暂停片段` 调用 `pause()`，ended、Segment 切换、content 变化和 unmount 会停止播放并释放 Blob URL。Transcript 只显示 `segment.md` 正文的真实文本；缺失或空 transcript 显示空态，不生成 mock 文本。
- Finalized audio SegmentSupplement content renderer flow：Memory detail loaded、selected Segment 存在且 selected Segment 有 finalized supplements 时，Memory Studio 在 `转录` 旁边为每个 finalized SegmentSupplement 显示独立 content tab。用户切换到某个 supplement tab 后，只有该 active audio SegmentSupplement panel 挂载，并使用 `['workspace', 'segment-supplement-content', workspaceId, memoryId, segmentId, supplementId]` Query key 调用 `workspace:readFinalizedAudioSegmentSupplement`。Renderer 只接受 requestId、workspaceId、memoryId、segmentId 和 supplementId 与当前 Query 匹配的 response；不匹配视为 stale response。Query 成功后，Memory Studio selected Segment resource cache 用 audio bytes 创建 local Blob URL 并缓存 waveform decode 结果，supplement player 只拥有播放进度、播放态和 scrub 状态。播放/暂停/seek 与 selected Segment playback 使用同一 Waveform UI 语义；播放行下方复用 feature-local `SegmentTranscriptView` 展示该 supplement 自身 transcript。pending 文案使用 `正在载入补充录音内容。`；读取失败时在 transcript 区显示 `补充录音转录加载失败，请重试。`；缺失或空 transcript 显示 `这段补充录音还没有转录。`，不生成 mock 文本。Supplement player unmount 时如果正在播放则先 pause；Blob URL 由 selected Segment resource cache 在 content request 刷新、supplement 从 selected Segment projection 消失、selected Segment 切换、Memory 切换、workspace handle 切换或 Memory Studio unmount 时释放。
- Segment content tab and supplement menu renderer flow：Memory Studio 的内容 tab 属于 selected Segment 的 feature-local UI state；audio Segment 始终显示 `转录` tab，selected Segment 的每个 finalized SegmentSupplement 都作为独立 tab 显示，tab label 使用 supplement title。同一 selected Segment 新增 finalized supplement 时，active tab 自动切到新增 supplement id；如果当前 active supplement 从 projection 消失，renderer 在同一 render 中 fallback 到 `转录` tab 和 transcript panel，随后用 effect 收敛本地 state。切换 Segment 或切换 Memory 时 active tab 回到 `转录`。内容 tab 的 id、panel id 和 keyboard selection 来自同一份 selected Segment content tab model；左右箭头、Home 和 End 直接选择 model value，不用 DOM query 或 click side effect 驱动状态。Supplement tab 的 More button 是 tab button 的 sibling DropdownMenu trigger；More visual reveal 只来自 owner tab item 的 CSS `group-hover` 和 More trigger 自身 `focus-visible`。Active selection 只让当前 More 进入键盘路径，不让它视觉常驻展开；DropdownMenu open 也不改变视觉折叠。非 active 且未 hover 的 More 同步收束 pointer events、tabIndex 和 `aria-hidden`；active 但未 hover/focus 的 More 仍保持 `max-width: 0` 和 `opacity: 0`。DropdownMenu 关闭时如果 trigger 已隐藏，焦点回到对应 tab。当前 More 菜单使用 `SegmentSupplementActionsMenu`，承载用默认应用打开、在访达中显示、复制路径、重命名和删除 intent；reorder 继续使用同一个 supplement tab 操作边界。`笔记`、`视频` 和 `图片` 不作为常驻禁用 tab 渲染，也不触发 IPC、Query、文件写入或 mock 内容。内容 tab 行右侧的 `+` 是 selected Segment 的 SegmentSupplement 菜单入口，不属于 tab 本身；菜单 open 状态只存在于 `MemoryStudio` component state，切换 Segment 或切换 Memory 时关闭。当前菜单只暴露录音补充项；选择后打开 `segment-supplement` recording target，显式携带 parent `memoryId` 和 `segmentId`。
- SegmentSupplement content resource flow：SegmentSupplement content Query 使用 TanStack Query 缓存 IPC response；Memory Studio 另有 selected Segment scope 的 feature-local audio resource cache，按 `workspaceHandle/workspaceId/memoryId/segmentId/supplementId/requestId/audioByteLength` 缓存 Blob URL 和 waveform decode 结果。用户在同一 Segment 内 A→B→A 切换 supplement tab 时不重复 IPC 读取、不重复创建 Blob URL，也不重复 waveform decode；同一 supplement 收到新的 content requestId 时先释放旧 resource；supplement 从当前 selected Segment projection 消失时释放对应 resource；切换 selected Segment、切换 Memory、workspace handle 改变或 Memory Studio unmount 时统一 revoke cache 中的 Blob URL。
- Segment rename renderer flow：Segment card More 菜单的 `重命名` 打开 `SegmentRenameDialog`，Dialog 复用 title form 组件并持有提交前 title draft。提交后 renderer 立即关闭 Dialog、更新 Memory detail cache 和 selected Segment focus，并调用 `workspace:updateSegmentTitle` 持久化。main 用 matching Memory 目录下合法 finalized Segment 文件空间节点校验和定位 Segment；写入成功后同步 `segment.md` title frontmatter 并刷新 index。成功 response 再同步 Workspace snapshot cache、Memory detail cache、当前 session Memory summary 和 selected Segment focus；失败时只在当前 Segment title 仍是本次提交值时回滚，并使用 root toast 显示可见错误。Segment rename 不创建新 Segment、不改变 `segmentId`、不读取或返回 raw path。
- SegmentSupplement rename renderer/main flow：SegmentSupplement tab More 菜单的 `重命名` 打开 `SegmentSupplementRenameDialog`，Dialog 复用 title form 组件并持有提交前 title draft。提交后 renderer 立即关闭 Dialog、只更新 Memory detail cache 中 parent Segment 的目标 supplement title，并保持 selected Segment focus；同时调用 `workspace:updateSegmentSupplementTitle` 持久化。main 先确认 request workspace 与 handle workspace 匹配，再用 matching Memory + parent Segment 下合法 finalized SegmentSupplement 文件空间节点校验和定位 supplement；命名上匹配 `supplementId` 或 `<supplementId>--*` 的 unsafe candidate 直接返回 unsafe typed error，不 fallback 到其它目录。Rename 把目录 basename 改为 `<supplementId>--<title>` 并写入 `supplement.md` title frontmatter，不改变 `updatedAt`，随后刷新父 Memory index entry。Renderer 在提交时捕获当前 `workspaceHandle`；await 返回后若当前 session 已不是同一 handle，则旧 response 不得 rollback、toast、写 cache、合并 session 或设置 focus intent。成功 response 同步 Workspace snapshot cache、Memory detail cache、当前 session Memory summary 和 selected Segment focus；普通失败时只在当前 supplement title 仍是本次提交值时回滚，并使用 root toast 显示可见错误。若 main 返回 `dataRetention: "file-written-index-stale"`，说明目录和 Markdown title 已写出但父 Memory index stale，renderer 保持 optimistic title、不做本地回滚，并显示 root toast。SegmentSupplement rename 不创建新 supplement、不改变 `supplementId`、不读取或返回 raw path。
- Segment delete renderer flow：Segment card More 菜单的 `删除` 打开 `SegmentDeleteDialog`，Dialog 使用 AlertDialog 危险确认。确认后 renderer 立即关闭 Dialog，从 Workspace snapshot cache、Memory detail cache 和当前 session projection 中移除该 Segment；Memory Studio 的 selected Segment fallback 到剩余第一条 Segment 或空态。App 同时显示可撤销 Sonner toast，10 秒 duration 是唯一 grace-period clock，底部进度条跟随该 duration；toast 的 `恢复` action 使用统一 icon+文字 action 结构，不显示 close button、不可手动 dismiss，只把目标 Segment 重新插回当前投影，并用一次性 Segment focus intent 选中恢复 Segment，不调用 main restore。Grace period 内的 Workspace refresh 必须保持该 pending delete projection，不得把尚未真实删除的 Segment 从文件真源带回 UI；同次 refresh 仍会 invalidate 非目标 Memory/Segment/Supplement content queries。toast 自动关闭时 App 只有在 workspace session 仍匹配原 handle 时才调用 `workspace:deleteSegment`；成功 response 合并父 Memory summary，并移除该 Segment 的 `segment-content` Query 和该 Segment 下所有 `segment-supplement-content` Query prefix；失败时用同一局部恢复路径并显示 root error toast。录音流程打开时 Segment 删除入口不可达。
- Segment delete/restore main flow：`workspace:deleteSegment` 和 `workspace:restoreDeletedSegment` 都必须在当前 handle owner、workspace root identity、`.reo` identity 和 lock usability 有效时执行。Delete/restore 正向 move 和 index-failure rollback move 都必须携带 destructive source read 得到的 Segment directory identity；rollback 使用本次已移动的 leaf name 和 directory identity，不按 Segment id 重新扫描 active/trash candidates，并在解析 parents、move 和 index rebuild 前后继续消费 workspace usability。验证后 active 或 trash source 被替换时不得移动替换目录，并返回 unsafe typed error 或 `file-written-index-stale` data retention；rollback 前 lock 已失效时不得继续搬文件。成功 move 后只扫描一次 active finalized Segment file truth，并用同一结果更新父 Memory summary 和 `.reo/index.json` entry；restore 还从同一结果投影 restored Segment。Restore parent Memory 是否存在直接由安全目录读取判定，不先做 exists 预检；restore 不做恢复前 active Segment candidate scan，同 id active duplicate 由 move 后的唯一一次 active Segment file-truth scan 检出并回滚。Delete move 保留 Segment 目录下的 `supplements/` 子树；restore 只有 parent Memory 仍存在时才把 trash Segment 移回 active tree。Rollback 失败时保留文件真源，不向 renderer 报告已完成删除或恢复。
- SegmentSupplement delete/restore renderer flow：SegmentSupplement tab More 菜单的 `删除` 打开 `SegmentSupplementDeleteDialog`，Dialog 使用同一危险确认结构。确认后 App 立即调用 `workspace:deleteSegmentSupplement`；成功 response 合并 Workspace snapshot cache、Memory detail cache 和当前 session projection，并移除该 supplement 的 exact `segment-supplement-content` Query。若被删 supplement 是当前 active content tab，Memory Studio fallback 到 `转录` tab。删除成功后显示统一 undo toast，`恢复` action 调用 `workspace:restoreDeletedSegmentSupplement`，成功 response 合并父 Memory summary、parent Segment projection 和当前 session projection，让 supplement tab 重新出现在同一 parent Segment 下；旧 workspace handle 或 session 不匹配时不应用 in-flight delete/restore response。若 delete 返回 `dataRetention: "file-written-index-stale"`，renderer 按 supplement 目录已进入 trash 处理，移除 parent Segment 中的 supplement 投影、清理 exact supplement content Query、关闭删除 Dialog，显示错误 toast，并仍显示同一 `已删除补充内容` undo toast；该恢复 action 使用 `restoreToken` 重新调用真实 restore。若 restore 返回同一 retention，renderer 按 supplement 目录已移回 active tree 处理，把保存的 supplement 投影回 parent Segment，并显示错误 toast。进入新 workspace session 或关闭当前 session 成功时，App 清理所有 workspace-scoped Dialog target 和 Segment focus intent。
- SegmentSupplement delete/restore main flow：`workspace:deleteSegmentSupplement` 和 `workspace:restoreDeletedSegmentSupplement` 都必须在当前 handle owner、workspace root identity、`.reo` identity 和 lock usability 有效时执行。Delete 把 parent Segment 下 matching finalized SegmentSupplement 目录移入 `.reo/trash/supplements/`，保留 `audio.webm` 和 `supplement.md`；restore 只有 parent Memory 与 parent Segment 仍存在时才把 trash supplement 移回 parent Segment `supplements/`。Parent missing 返回 `ERR_SEGMENT_SUPPLEMENT_RESTORE_PARENT_MISSING`，trash 目录保持可恢复。成功 delete/restore 都刷新父 Memory index entry 的 `supplementCount` 并返回 parent Segment projection，不返回 raw path 或 handle。
- SegmentSupplement recording renderer flow：录音 overlay 在 `segment-supplement` target 下复用同一 recording surface 和 microphone intent，但 draft 创建、append、finalize 和 discard 分别调用 `workspace:createSegmentSupplementRecordingDraft`、`workspace:appendSegmentSupplementRecordingAudioChunk`、`workspace:finalizeSegmentSupplementRecordingDraft` 和 `workspace:discardSegmentSupplementRecordingDraft`。Draft id 是 supplement id，不是同级 Segment id。Finalize request 必须携带 `workspaceId`、parent `memoryId`、parent `segmentId`、`supplementId`、title 和 duration；默认 title 是所选 Segment 内的 `补充录音N`。成功 response 返回 memory summary、parent segment projection 和 supplement projection。Renderer 只用 memory summary 更新 Workspace snapshot，只用 parent segment projection 更新 Memory detail cache，对 supplement projection 不插入顶层 Segment strip。SegmentSupplement 录音 marker 使用 `targetKind: "segment-supplement"`、`parentSegmentId` 和 supplement id；保存恢复时调用 supplement finalize，放弃恢复时调用 supplement discard。当前 SegmentSupplement 恢复只提供保存或放弃，不提供回听、继续或替换，因为 draft audio preview IPC 只定义了 Segment draft read。
- Workspace Stage recording target flow：底部 `录音` 优先使用当前 Memory context 直接打开 existing-memory recording target；如果当前 snapshot 没有 Memory，则先打开 `MemoryCreateDialog`，创建成功后使用返回的 `memoryId` 打开 existing-memory recording target。创建失败停留在命名弹层，recording overlay 不打开，mic intent、draft directory、recording IPC 和 MediaRecorder 都不得启动。
- Memory create main flow：`workspace:createMemory` 校验 sender、handle ownership 和 request schema 后，由 main process 生成稳定 `memoryId`，在 `memories/` 下 no-replace 创建空 Memory 容器，写入 `memory.md` 和 `.reo/objects/memories/<memoryId>.json`，刷新 `.reo/index.json`，返回 Memory summary。Memory directory 已存在、`memories/` 不安全或 index 刷新失败时返回 `ERR_MEMORY_CREATE_FAILED`；如果已经写入 Memory 文件真源，普通失败必须删除本次刚创建的目录并重建 index，避免 index 指向缺失目录。Delayed lock usability 返回 lock lost 时必须停止后续写入、清理和 index 重写，直接返回 lock lost。
- Recording lifecycle 覆盖 idle、acquiring-permission、recording、paused、finalizing 和 failed；mic intent 拒绝、permission denial、media start failure 或 append failure 从忙碌状态进入 failed，discard 当前未完成 draft，不创建 finalized audio segment。Finalize failure 从 finalizing 进入 failed，并按 main error envelope 的 data retention 保留或清理 draft。idle 收到 late failed event 必须 no-op。
- Recording overlay 忙碌状态包括 acquiring-permission、recording、paused 和 finalizing；Drawer/Vaul close、Escape 和 overlay dismiss 必须阻止这些状态误关。App route 层把存在 recording target 视为录音流程正在占用当前工作区，阻止 sidebar 和 MemoryRail 引发的工作区切换、workspace release、移除、创建叠层或 current Memory 切换；overlay 同时把 busy close-blocked state 上报给 App，App 只在忙碌录音态通过 `beforeunload` 阻止原生窗口关闭或 reload，准备态不阻止原生关闭。
- Recording overlay target model：loaded workspace frame 的 `录音` 只打开显式 target；已有当前 Memory 时是 `existing-memory`，没有 Memory 时先完成 Workspace Stage recording target flow，再用新 `memoryId` 打开 `existing-memory`。普通录音默认 title 是目标 Memory 内的 `录音N`；恢复录音使用 recovery marker 中保存的 title。Finalize request 必须携带 `memoryId`，main 只追加 audio segment，不从 recording 隐式创建 Memory。
- Recording overlay session model：`开始录音` 先完成 microphone intent，再创建 draft，最后启动 browser MediaRecorder adapter。只有当前 session 的 MediaRecorder controller ready 后 UI 才进入 recording；stale session、组件 unmount、workspace handle 切换或 start failure 都只清理匹配 pending intent，并丢弃不再有效的 draft。
- Recording capture model：MediaRecorder adapter 是 durable WebM/Opus audio chunk 真源，append queue 串行写入 draft；同一 stream 的 analyser 只提供 waveform level samples，AudioWorklet PCM 只服务 live ASR。ASR start/send/finish 失败不影响 durable capture，只关闭 live ASR、标记 completion backfill 并给出可见错误。
- Recording pause and preview model：pause 会停止采集时钟、麦克风音轨、PCM 输入、level pump 和当前 ASR 输入，并把 cursor 放到当前有效末尾。Paused playback preview 只由当前 session 的完整 MediaRecorder chunk 前缀生成；preview 状态绑定 recording session、Blob source 和 chunk count，ready 事件必须匹配当前 source。已有有效 chunk 前缀时，flush timeout 不阻塞先准备可播放前缀，迟到 chunk 到达后重建 preview source。
- Recording seek model：暂停态定位只通过 waveform click 或 waveform pointerdown 创建的 scrub session 完成；未由 waveform pointerdown 创建的 pointermove 不得改变 cursor 或 audio currentTime。播放从合法预览起点开始：cursor 在末尾或回放 ended 后再次播放时从 0 开始，cursor 在中间时从 cursor 开始；ended 后 cursor 回到末尾。
- Recording paused action model：暂停态只暴露当前真实能力存在的 action。controller 仍存在且 cursor 在末尾时可以 continue；controller 仍存在、target 是 `existing-memory` 且 cursor 位于中间时可以 replace；recovered draft 和 `segment-supplement` 中段暂停不暴露 continue 或 replace。
- Recording replacement model：replace 单击立即进入 replacement flow，不用 toast 或二次确认。cursor 大于 0 时 renderer 先按实际 durable chunk 边界保留前缀，再用同一 MediaRecorder controller 从该位置继续；cursor 为 0 时创建新的 recording session 和 controller 从头采集。没有 word-level timing 时，cursor 落在 transcript segment 中间会丢弃重叠 segment 并标记 completion backfill，不按比例裁剪文本。rollback 只在原 session 仍有效时恢复旧 draft；session 已失效时只清理新 draft。
- Recording finish model：用户点击完成后 visible recording surface 立即关闭，App 保留隐藏 lifecycle owner。后台 stop flow 停止 recorder、等待最后 data chunk、清空 append queue、finish ASR 并 finalize durable audio；最终转写失败只影响 transcript 保存，不回滚 durable audio。Finalize audio 成功后先写完整 finalized audio Segment projection 或 finalized SegmentSupplement projection recovery marker，再把 transcript 写入对应 finalized `segment.md` 或 `supplement.md` 正文；transcript save 失败时后续恢复只补 transcript，不重复 finalize。
- Recording close model：非忙碌状态关闭 overlay 会清空 renderer-local draft、error、elapsed、cursor、waveform samples、playback URL 和 machine state，并忽略已关闭 session 的 transcript save/ASR 结果。下一次打开必须从 idle ready state 开始。
- 当前 recording boundary flow：录音有效时长小于 2 秒时，首次 stop 只给出中性短录音提示并保持当前录音数据，第二次 stop 才允许 finalize；持续低音量输入达到 15 秒时只显示一次中性提示，capture、append queue 和 ASR 输入继续；单次录音到 55 分钟显示接近上限提示，达到 60 分钟自动执行 pause flow 并把 cursor 设置到当前有效末尾；替换单击直接进入 replacement flow，不使用覆盖提示 toast 或二次确认。中段 replacement 复用当前 MediaRecorder controller，不创建新的 microphone intent；起点 replacement 使用新的 recording session 和 controller。replacement draft 创建、retained prefix copy 或 fresh capture restart 失败必须保留原 active draft，不能误导按钮状态或丢失原录音。
- cursor 大于 0 的替换 retained audio copy 由 main process 的 `workspace:cloneRecordingDraftPrefix` 完成：renderer 先等待 append queue 静止并创建 target draft，再请求 main 从 source draft 复制 cursor 前有效 byte prefix 到 target draft。Main 必须校验 source/target metadata、source retained byte length、target 空音频、同 draft write lock 和 workspace containment；复制循环中每个 read/write chunk、target fsync 和 metadata 写入前都必须重新消费 handle lock usability。copy 失败或 lock lost 必须把 target `audio.webm` 回滚为空并 fsync，renderer rollback 必须恢复旧 draft、captured chunks、PCM chunks、waveform、duration、cursor、revision、timeline 和 transcript draft。cursor 为 0 的替换不得调用 prefix clone，不得把无初始化 header 的旧 WebM 后段作为新音频；必须从新的 MediaRecorder controller 和新 recording session 开始。
- 当前 recording abnormal recovery flow：renderer 在 recording draft 创建后写入按 `workspaceId` 隔离的 local recovery marker，记录当前 `memoryId`、`segmentId`、title、有效 duration、append `nextSequence`、recording session、revision、waveform samples、audio chunk time/byte map、safe audio byte length 和当前有效 transcript segments；SegmentSupplement marker 额外记录 `targetKind: "segment-supplement"` 和 parent `segmentId`。录音中 duration 至少每秒更新一次，首个 MediaRecorder chunk 必须写入恢复快照，后续 chunk map 按节流更新，ASR segment、pause/stop 前和 finalized audio 已保存但 transcript 尚未保存时会刷新恢复快照；append ack 不逐次写 localStorage，恢复继续检查时以 draft metadata 读取真实 `nextSequence`。marker 超出预算时可以裁剪 waveform 或压缩 audio chunk map，但 transcript 文本不得静默丢弃；必要时 transcript markdown 写入同 workspace 的 renderer recovery sidecar，marker 保存引用和 segment 校验。正常 finalize、主动 discard、replacement 丢弃旧 draft、组件 unmount 丢弃 draft 或关闭清理会删除匹配 marker 和 transcript sidecar。重新打开同一 workspace 后，App 只在 marker 的 `workspaceId` 匹配且 `memoryId` 仍存在于当前 snapshot 时显示“未完成录音”恢复对话框；用户选择保存时使用当前 `workspaceHandle`、marker 的 `memoryId`、`segmentId`、title 和 duration 调用对应 finalize IPC：普通录音调用 `workspace:finalizeRecordingDraft`，补充录音调用 `workspace:finalizeSegmentSupplementRecordingDraft` 并携带 parent `segmentId`。成功后合并 Memory summary 和完整 finalized projection；如果 transcript 保存失败，App 将 finalized audio Segment projection 或 finalized supplement projection 和最新 transcript snapshot 写回 marker 并保留恢复提示，后续保存重试只补对应 finalized `segment.md` 或 `supplement.md` 正文，不把 `ERR_RECORDING_NOT_FOUND` 当作成功；audio finalize 和 transcript save 都完成后才清除 marker 并保持当前 Memory context。用户选择放弃时调用对应 discard IPC，成功后清除 marker；用户选择继续检查时 App 打开该 Memory 的 recording overlay，overlay 只有在 marker 存在 audio chunk byte map 时才通过 `workspace:readRecordingDraftAudio` 读取未完成 draft audio，读取上限来自 marker 的 chunk byte map，并按 marker 的 chunk map 还原暂停态 chunk 前缀、waveform、cursor、timeline 和 revision；如果 marker 只有 transcript sidecar markdown 而没有 transcript segments，overlay 必须把该 markdown 作为待保存 transcript draft。没有 marker-derived chunk map 时不得用固定大上限读取 draft audio，只允许保存或放弃。恢复后的暂停态可以直接完成保存、回听检查或放弃；由于没有原 MediaRecorder controller，继续录制和中段替换不会跨会话拼接 WebM，UI 不暴露继续或替换 action。恢复失败使用 root toast，marker 保留以便重试。
- 当前豆包流式语音识别完成 main 侧 live session helper 和 renderer subscription 边界：recording transcription start 先通过 `resolveVoiceSettings()` 读取 application-scoped voice settings 快照；`enabled=false` 时返回 `accepted: true` 和 `transcriptionMode: 'disabled'`，不创建 live session；`enabled=true` 且没有可解密 X-Api-Key 时返回 `ERR_RECORDING_TRANSCRIPTION_UNAVAILABLE`；`enabled=true` 且 X-Api-Key 存在时进入 live session。火山 header 由 main runtime 输入构造，使用单 `X-Api-Key` 加 `X-Api-Connect-Id` 和 `X-Api-Resource-Id`，资源 ID 是 `volc.seedasr.sauc.duration`，连接端点是 `wss://openspeech.bytedance.com/api/v3/sauc/bigmodel_async`，full request 使用 16kHz/16bit/mono raw PCM、`bigmodel`、`show_utterances`，audio-only request 使用 gzip 分包和正/负 sequence，server response 解析为带 `recordingSessionId` 和 `revisionId` 的 transcript segment；main `recordingTranscriptionSessions` 按 sender、workspaceHandle、recordingFlowSessionId 和 recordingSessionId 绑定 active session，替换或重新开始会关闭旧 entry，旧 revision audio 和旧异步 segment 返回都会被丢弃。Preload 只暴露显式 start/send/finish/close 方法和去除 `IpcRendererEvent` 的 `onRecordingTranscriptionEvent` callback；renderer 不持有已配置的 X-Api-Key 明文或火山鉴权 header。Renderer live PCM 发送必须通过按 recording session / flow / revision 绑定的有界串行队列，队列 drain 期间如果 live session 已关闭则不再调用 finish，并改走 completion backfill；live PCM send 返回 `accepted:false` 时也必须关闭 live ASR 并标记 completion backfill，避免已有部分 transcript 时尾部音频静默掉队。ASR `start` 初始连接失败时由 main registry 重试一次，临时失败不会向 renderer 发送 transcript error event；重试仍失败时返回脱敏错误。录音中的 live session 发生 transport error 或非预期 close 时，main registry 会用同一 recording session 和 revision 启动新 session，保留并回放最近 5 秒 16 kHz PCM buffer；如果 replay 起点落在某个 PCM chunk 内，main 会裁剪首个重叠 chunk，避免重复发送 replay 起点之前的音频；回放段的 transcript timestamp 继续映射到原录音时间线；重连成功不向 renderer 发送临时 error event，重连耗尽后才发送脱敏 error event，durable capture 继续。Renderer 同步缓存当前有效 16 kHz PCM chunks；如果完成时实时 transcript 为空，或当前 revision 已收到 ASR error / finish failure，会先 finalize durable audio，再用同一 `recordingSessionId` / `revisionId` 和 `completion-backfill` recording flow session 将缓存 PCM 重新发送给 main ASR session，等待最终包后保存 transcript；补转写失败只使用 root toast 告知，durable audio 已保存结果不回滚。ASR `finish` 发送 final audio frame 后必须等待服务端 last package，收到最终 segment 后才确认 closed；最终包超时、socket 提前关闭或 finish 失败时返回安全错误，renderer 用 root toast 告知并继续保存 durable audio。close transcription 在 sender/handle owner 匹配后允许在 workspace lock lost 时清理 active session；finish transcription 在 lock lost 时必须先关闭 owner-matched active ASR session，再返回 lock lost，避免无 UI owner 的 live socket 残留。
- Audio append 必须按 sequence 串行 ack；每个 recording draft 只允许 1 个 append 在途，超量进入 failed recoverable。当前进程创建的 active draft 可以跳过每个 append 的 durable finalized scan；记忆空间 close、window teardown 或 root runtime state 清理后必须移除该进程态，后续 append 重新检查 finalized truth；stale/manual draft append 必须先检查 durable finalized audio segment。存在 finalized truth 时返回 finalized error，即使 `.reo/drafts/segments/<segmentId>/` 中仍有 stale draft。Append 写 chunk 前必须确认 draft `audio.webm` 是记忆空间内普通文件，拒绝 symlink 或非文件 audio；metadata 写失败时必须 truncate audio 回 append 前 size 并 fsync，返回 `ERR_RECORDING_APPEND_FAILED` 与 `dataRetention: "draft-preserved"`，且不得推进 sequence、audio bytes 或写到记忆空间外。
- MediaRecorder chunk 转换失败、append 返回错误、append promise reject 或 media start 失败时，recording overlay 立即进入 failed，不调用 finalize，停止当前 recorder controller，discard 当前未完成 draft，并用 recording session token 忽略 stale recorder 后续 chunks；discard draft cleanup 必须重新验证 workspace containment，不能跟随 symlink parent 删除 记忆空间外目录；failed 状态允许用户重新开始一次新的 recording draft，retry 会清空 stale elapsed、transcript draft 和 timeline state。
- Finalized audio segment 不允许继续 append。Finalize 必须先复核 `memories/` root，再按 `memoryId` 解析 matching Memory 目录并复核 `segments/`，不能用递归 `mkdir` 跟随 symlink；创建 memory directory 和 segments directory 的目标解析后、实际 `mkdir` 前后都必须重新解析 workspace child containment，拒绝 ancestor symlink swap。随后在已验证 `segments/` parent 内使用相对 `mkdir` 创建 Reo-owned `.reo-finalizing-*` staging directory，不得用缓存 absolute path 创建 staging；staging 创建后写 marker、fsync、copy 和 expose 前都必须重新解析当前 staging directory；staging draft `segment.json` 读取必须使用 no-follow file handle。再通过完整 workspace child containment 重新解析 draft source，复核 `.reo/drafts/segments` ancestor 和 leaf draft directory 都不是 symlink 或非目录，复制前再次解析 draft source，只允许异步读取普通 draft `segment.json`、`audio.webm` 和 `transcript.md`，并写出 finalized `segment.md`、`audio.webm` 与 `.reo/objects/segments/<segmentId>.json`；未知文件必须失败并保留 draft；复制后 fsync staging tree。在 expose 前必须重新通过完整 containment 解析 target segment path，并把新 Segment 目录命名为 `<segmentId>--<title>`；最终 expose 在已验证 parent directory 内重验 target missing，创建 target directory 作为 no-replace reservation，再先搬迁 marker、后搬迁 staging 内容，parent identity 改变或 late-created target 必须失败，不能使用已缓存 absolute path 直接暴露；最后 fsync parent。
- Finalize cleanup 在删除 draft、staging、target 或未提交 memory 目录前后必须重新做 workspace containment；如果 cleanup path 在验证后变为 unsafe，main 必须失败或跳过删除，不得删除 记忆空间外文件。
- Finalize 在 staging copy 后、durable target expose 前必须按 draft metadata schema 重新校验复制结果，并确认 `workspaceId`、`segmentId` 和 `audioByteLength` 与当前请求及实际 audio file 一致；该校验失败属于 pre-expose failure，必须清理 staging、保留 draft、不得删除 draft 或写出 finalized truth。
- Finalize 在 staging 暴露前失败时必须删除 `.reo-finalizing-*` staging tree；如果这是新建 memory 且尚未写入 `memory.md` 和 Memory manifest，也必须删除该空 memory 目录，保证同一显式 `memoryId` 可以重试。
- Directory fsync 在不支持的平台或文件系统上按 best-effort 处理；`EACCES`、`EISDIR`、`EINVAL`、`ENOTSUP`、`EPERM` 不得让已成功写入的 atomic file 或 finalize transaction 误回滚。
- Finalize 使用 renderer active recording clock 计算的显式 duration 和实际 `audio.webm` 文件大小写入 `.reo/objects/segments/<segmentId>.json`、返回值和 `.reo/index.json`，三者的 `audioByteLength` 必须一致，且 manifest `objectType/memoryId/segmentId` 必须与目录 ownership 一致。`memory.md`、Memory manifest 写入和 previous-memory rollback 前必须重新解析 memory directory，不能复用 final rename 前缓存的 memory path。
- Memory Markdown、记忆空间 metadata、workspace index、`.reo/objects/*` manifests 和 finalized Segment Markdown 必须从记忆空间内普通文件读取；`.reo/workspace.json`、`.reo/index.json`、`.reo/objects/*/*.json`、`memory.md`、`segment.md` 和 `supplement.md` leaf 必须用 no-follow file handle 打开，symlink leaf 不进入 open、rebuild 或 lookup 投影。每个 metadata/index JSON 文件不得超过 1 MiB，schema 必须 strict，unknown 字段不进入当前合同。Segment delete/restore 的 destructive source read 对 unsafe path 是 strict：`segment.md`、对应 manifest、`audio.webm` 或 source directory 不安全时返回 unsafe typed error，不能降级成 not-found 或换路径 fallback。Finalized audio segment transcript save public request 必须携带 finalize response 返回的 `memoryId` 和 `segmentId`；main 直接解析 nested finalized segment directory，遇到 unsafe finalized Markdown/manifest、ownership mismatch 或 duplicate durable segment id 必须返回 typed not-found/error envelope，不能 fallback 到 stale draft。Finalized audio SegmentSupplement transcript save public request 必须携带 parent `memoryId`、parent `segmentId` 和 `supplementId`；main 直接解析 parent Segment 下 matching supplement 文件空间节点，遇到 unsafe supplement Markdown/manifest、ownership mismatch 或 stale basename 必须返回 typed error，不能 fallback 到 stale draft。markdown save 必须在当前 segment 或 supplement directory identity 内执行 atomic write，同一 root/memory/segment/supplement/file 进入 main process 写队列，避免 out-of-order save 旧内容覆盖新内容。
- Finalized audio file 必须是记忆空间内普通文件，不能是 symlink 或非文件；index rebuild、finalized lookup 和 finalized read 都必须拒绝 symlinked `audio.webm`。`workspace:readFinalizedAudioSegment` 只允许读取当前 handle 下的 explicit `memoryId` + `segmentId` finalized audio truth，返回 bounded audio bytes、audioByteLength 和 `segment.md` 正文文本，不返回 raw path；读取前后必须校验 handle usability、duplicate finalized segment id、manifest ownership、audio byte length 和 segment directory identity。`workspace:readFinalizedAudioSegmentSupplement` 只允许读取当前 handle 下 explicit `memoryId` + `segmentId` + `supplementId` 的 finalized supplement truth，返回 bounded audio bytes、audioByteLength 和 transcript（exists 与 text），不返回 raw path；读取前后必须通过 supplement manifest id 定位真实文件空间节点，并校验 parent segment directory identity、supplement manifest ownership、audio byte length、supplement directory identity 和 `supplement.md` transcript section。暂停态回听默认使用 renderer 当前录音会话持有的有效 chunk 前缀创建临时 Blob，异常恢复只通过 `workspace:readRecordingDraftAudio` 在 marker byte map 限制内读取未完成 draft audio 并按 marker chunk map 重建回听检查前缀，不跨 MediaRecorder session 继续或替换。`workspace:readRecordingDraftAudio` 与同 draft 的 append、finalize 和其他 read 互斥；finalize 开始前也必须拒绝 active draft audio read，读取后必须重新校验 metadata byte length 与读取上限，避免 append/read/finalize race 返回超出 marker 上限或删除正在恢复预览的 draft。
- Finalize 复制 draft 后只能把 draft `transcript.md` 转写为 finalized `segment.md` 正文；如果 draft 中已有普通 `transcript.md`，必须保留内容，不能用空正文覆盖。
- Finalize 更新 `.reo/index.json` 失败时，main process 删除 interrupted staging/target directory、恢复 Memory Markdown/manifest 并保留 draft，返回 `ERR_RECORDING_FINALIZE_FAILED` 和 `dataRetention: "draft-preserved"`；如果 target directory 已存在且不属于当前 finalize marker，不得删除。若 delayed handle usability 在 finalize 内部返回 lock lost，必须立即返回 lock lost，不得继续执行 stale handle cleanup、rollback 或 index 写入；后续 recovery 由新 owner 在重新获取 lock 后处理 marker/staging。
- Finalize 在 index 已刷新后又因 draft cleanup 前错误需要 rollback 时，必须删除 interrupted staging/target directory、恢复或删除 Memory Markdown/manifest，并重新刷新 `.reo/index.json`，避免 projection 指向已回滚的 recording。
- Finalize 成功时先删除 draft 并 fsync draft parent，再重新解析 durable target directory、复核 containment、清除 durable target marker；如果 draft 删除前失败，返回 `dataRetention: "draft-preserved"`；如果 draft 已安全缺失，按 cleanup 已完成处理，不得回滚已写入的 durable audio segment；如果 draft 已删除但 draft parent fsync、target 复核或 marker 清除失败，返回 `dataRetention: "durable-marker-recovery-required"`，不能把该 finalized audio segment 当作已完整收口。删除 draft 后、draft parent fsync 和 marker unlink 前必须重新消费 delayed handle usability；中途 lock lost 时保留 durable marker，让后续新 owner recovery 收口。Recovery 看到 marker-bearing valid finalized audio segment 或 SegmentSupplement 时也必须先删除对应 stale draft、确认 draft parent fsync、重新解析 target directory，再清除 marker；如果 draft 已安全缺失，按 cleanup 已完成处理，继续 fsync parent 并清 marker，避免 stale draft 影子读取或 durable marker 永久保留。open 调用 recovery 时，staging/partial 删除、draft cleanup、draft parent fsync、marker unlink、orphan cleanup 和 memory repair 写入前必须消费当前 lock/root/`.reo` usability；中途 lock lost 时停止 recovery 写操作并返回 lock lost。Recovery 只有在当前 Memory 下已经发现 finalize marker、staging 或 supplement recovery work 时才协调合法 finalized Segment 文件空间节点；没有 recovery marker 时不得为了修复投影全量读取 Segment manifest。Memory detail、index rebuild 和目标写入都以文件空间节点真源为准。Recovery 遇到缺失 `memory.md` 或 Memory manifest 但带 marker 且含 `segment.md` 或 `audio.webm` 的 finalized segment payload，或遇到无法确认属于 Reo staging 的 markerless durable payload，必须 fail-open 保留；删除 staging/partial segment 后，只删除没有 Memory Markdown/manifest、没有用户 payload 且仅剩空 `segments/` 结构的 metadata-less memory directory。
- 同一 `memoryId` 的 create、append、Memory title update、Segment title update 和 Segment delete/restore 通过 main process memory write lock 串行保护；并发 append 中未获得锁的一方返回 `ERR_RECORDING_FINALIZE_FAILED` 和 `dataRetention: "draft-preserved"`，并发 title update 中未获得锁的一方返回错误信封并保留既有文件真源。
- Open 会读取合法 `.reo/index.json` 作为启动 cache，并用该 cache 在获取 lock、恢复 finalize marker 和补齐托管目录后返回 ready snapshot；不在合法 index 存在时为了比较而全量扫描 Memory 文件树。`.reo/index.json` 损坏、丢失或 symlink leaf 时才从文件真源 rebuild。Active renderer 在 ready 后立即执行一次静默 `workspace:readWorkspaceSnapshot`，之后只在 document visibility 变为 visible 时刷新，从文件真源协调外部合法修改。Scan 只从 schema、ownership 和 manifest/audio 一致的 finalized audio segments 与 supplements 协调 memory summary。Scan 必须绑定 `memories/` root identity，scan 前、scan 后或持久化前 identity 改变时失败并保留原 index；如果 `memories/` 存在但不可读取，需要 scan 的 open rebuild 或 snapshot refresh 必须失败并保留原 index，不能把 read error 协调为空记忆空间。Memory 和 Segment 扫描以 `.reo/objects/*` manifest id 建立身份，目录 basename 只作为 title 投影；Segment 归属来自 Segment 文件空间节点的 manifest ownership。Workspace title update、Memory title update 和 Segment title update 在合法 index 存在时不为投影 response 全量扫描 Memory 文件树。Transcript save、Segment finalize 和 SegmentSupplement finalize 只刷新所属 memory 的 index entry，避免每次写入重扫全 workspace；同一 workspace 的全量 index replace 和 index entry refresh 进入同一个 main process write queue。全量 replace 在队列内重新读取当前 read model 后再写入；单条 refresh 在队列内部重新读取当前 memory summary，再按顺序读取当前 index、合并单条 memory summary、atomic 写回。
- Recovery 扫描当前 matching Memory 目录的 `segments/` 前必须重新做 workspace containment 和 symlink guard；如果 `segments` 本身或 Segment leaf directory 是 symlink，不能跟随、不能删除 symlink target。没有 marker、staging 或 supplement recovery work 时，Recovery 只做轻量 marker/staging 检查，不读取所有 finalized Segment manifest，也不改变合法 Segment 文件空间节点本身。
- 如果进程在 finalized metadata 写入后、`.reo/index.json` 写入前崩溃，后续 active snapshot refresh 会从记忆空间文件真源恢复 `.reo/index.json` 的 memory summary，并让 Workspace snapshot 的 `memories[]` 投影重新可见；如果 `.reo/index.json` 缺失、损坏或 unsafe，open 会先 rebuild。
- Metadata、index 和 transcript 的 atomic write 先异步写入并 fsync temp file，再在已验证 parent directory 内 rename；temp open、data write、rename 与 parent fsync 前后复核 parent directory identity。replace write 覆盖已有 target 前必须保留同目录 backup，backup 删除前的后置 validation 或 fsync 失败时恢复旧 target，不能删除唯一副本；backup 已删除后的 cleanup durability failure 属于 best-effort，不得把已提交 target 误报为失败；`AGENTS.md` no-replace write 在 temp file fsync 后 hard-link 到目标 path。当前 Node API 没有 fd-relative `renameat/linkat`，final commit 使用短同步 critical section 临时进入已验证 parent directory；该窗口内不得 `await`，也不得调用依赖 `process.cwd()` 的外部逻辑。若 commit 后发现 parent path identity 改变，必须删除本次暴露的 target；replace write 还必须恢复旧 target。parent directory fsync 成功时提供更强持久性，遇到当前平台或挂载点不支持 directory fsync 的错误时不得在 rename 或 hard link 后误报写入失败。
- `updateWorkspaceIndex` 在 update 函数成功前不会持久化 open/rebuild 协调结果；失败路径不改变已有合法 index。
- Renderer 点击完成后立即关闭 visible recording surface，但 RecordingOverlay lifecycle owner 必须继续挂载到最后一个 append ack、durable finalize、completion backfill、transcript save 和 recovery marker 收口完成；后台收口失败时必须重新打开 recording surface 到失败恢复态，不得留下隐藏阻塞的录音 owner。
- Finalize 必须等待最后一个 append ack，不得 duplicate stop 提前 finalize。
- Transcript save failure 使用 root toast 显示，durable audio finalize 结果保留，main atomic write 保留 previous disk file。
- Paused draft playback 不读取 finalized audio IPC，只用当前 renderer 已捕获的有效 MediaRecorder chunk 前缀创建临时 Blob URL；pause 会 request/flush 当前 MediaRecorder dataavailable chunk，已有 chunk 前缀时不等待迟到 flush 才开始准备回听，隐藏 audio 完成可播放预加载前播放按钮保持禁用；新 chunk 到达、替换、完成、关闭或 unmount 时必须停止回听并释放旧临时 URL。
- Memory space write 使用 single-writer lock；第二个 Reo writer 返回 locked error。带 owner 进程已退出、进程启动指纹不匹配、缺失 owner file、损坏 owner file 或 symlinked owner file 的 stale lock directory 可在下一次 acquire 时被替换；不含启动指纹的 owner file 通过 owner file mtime 与当前 pid 启动时间判断 PID 复用；无法确认 owner 是否仍是同一活进程时不做 silent override。

## 时序规则

- 多步骤 write 必须有 transaction boundary。
- 改变 cached data 的 mutation 必须有 invalidation 或 rollback。
- TanStack Query mutation 必须定义 query key、invalidation scope、pending/error state 和 rollback 策略。
- Zustand persist 必须定义 storage、version、partialize/migrate 和 user-visible recovery。
- Form submit 必须定义 validation timing、submit owner、failure behavior 和 reset behavior。
- Background task 必须有 owner、trigger、cancellation、retry、error reporting。
- Diagnostic/background error flow 必须有 owner、trigger、redaction、retention、retry 和 failure behavior。
- Session/auth change 必须写明 caller、persistence effect、renderer visibility、recovery behavior。
- Better Auth Electron flow 必须先定义 system-browser request、PKCE/state、callback exchange、session persistence、renderer visibility、sign-out、error 和 recovery。
- Packaging flow 必须先定义 build、package、sign、notarize、make、publish、verify 的顺序和 failure behavior。
- Update flow 必须先定义 trigger、channel、metadata source、download/install timing、cancellation、retry、rollback、error reporting 和 renderer visibility。
- Concurrency-sensitive flow 必须写 race condition analysis。

## 禁止

- 禁止 generic event runtime。
- 禁止没有当前产品需求的 mailbox 或 queue runtime。
- 禁止没有 observability 的 hidden background job。
- 禁止没有 signed packaged app 和 release metadata 的 updater polling。
- 禁止没有 rollback 的 optimistic UI path。
- 禁止依赖未记录 timing assumption 的 async flow。
